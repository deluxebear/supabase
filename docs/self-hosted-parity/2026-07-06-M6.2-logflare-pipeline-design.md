## §1 Problem and goal

Since M2.1 the per-ref analytics plumbing exists (`getAnalyticsTarget` strict
no-fallback, 3 analytics routes, `retrieveAnalyticsData`), but the stack-side
analytics components (`docker-compose.logs.yml`: Logflare + vector) had never been
deployed and the data path had never been live-verified. The home usage charts and
every Logflare-backed observability surface honestly fail today. M6.2:

1. Verifies and documents the stack-side analytics bring-up (operator runbook).
2. Makes the Studio surfaces actually render real data against a stock
   **Logflare postgres-backend** deployment — which the spike showed requires
   SQL adaptation, not just deployment.
3. Adds the real edge-functions probe (M6.0 D4 debt) and cleans the two M6.0
   realtime minors.
4. Folds in the M6.1 follow-up bundle.

## §2 Spike facts (live-verified 2026-07-05/06 on the dev stack)

Deployment:

- `docker compose -f docker-compose.yml -f docker-compose.platform.yml -f
docker-compose.logs.yml up -d analytics vector` works **as-is**: env complete
  (`LOGFLARE_PUBLIC/PRIVATE_ACCESS_TOKEN` ship in `docker/.env`, placeholder values),
  `_supabase` db + `_analytics` schema pre-exist, docker socket configured. Zero
  disturbance to the 14 running services (targeted `up`).
- The **only gap**: the compose file ships the `4000:4000` ports mapping commented
  out — a host-process Studio (dev server, or any Studio reaching the stack from
  outside the compose network) cannot reach `http://localhost:4000` without it.
- Logflare 1.43.1 boots healthy; `LOGFLARE_SUPABASE_MODE` seeds 9 sources
  (`cloudflare.logs.prod`, `gotrue.logs.prod`, `postgres.logs`, `realtime.logs.prod`,
  `storage.logs.prod.2`, `postgREST.logs.prod`, `pgbouncer.logs.prod`,
  `deno-relay-logs`, `deno-subhosting-events`) and **only 2 endpoints**:
  `logs.all` (sandboxable) and `usage.api-counts`.

Data path (first live verification ever):

- kong traffic → vector → Logflare → `GET /api/endpoints/query/logs.all?project=default`
  returns real events with metadata. Multi-source ingestion confirmed (kong + gotrue;
  per-service volume depends on what each service writes to stdout — postgREST logs
  nothing per-request by default; pipeline itself is sound).
- `?project=default` assumption (M2.1) holds: vector tags every event
  `project=default`; the seeded endpoint SQL filters on it.

Logflare postgres-backend compatibility (the decisive findings):

- **`usage.api-counts` is broken server-side**: its seeded BigQuery SQL fails
  BQ→PG translation (`cannot subscript type text`, pg_code 42804, parameter-independent).
  Stock upstream self-hosted has the same failure. → home usage charts cannot be lit
  by the native endpoint.
- Unknown endpoint names (`logs.all.otel`, `auth.metrics`, `service-health`,
  `functions.combined-stats`) → Logflare **401** `{"error":"Unauthorized"}` (not 404).
- **`logs.all` sandbox SQL is the reliable channel**: all 9 service tables are exposed
  as CTEs (`edge_logs`, `postgres_logs`, `auth_logs`, `function_edge_logs`, …).
  Translation rules pinned by bisection:
  - ✅ `countif`, `regexp_contains`, multi-level `cross join unnest`, field access on
    unnested records, `timestamp_trunc`, `group by <ordinal>`, `limit/order`.
  - ❌ `cast(… as datetime)` (`type "datetime" does not exist`), `group by <novel alias>`.
  - ⚠️ `group by <alias shadowing a column>` is **accepted but silently groups by the
    raw column** (fan-out, wrong results, no error) — variants must use ordinals.
  - Timestamps come back as **epoch microseconds** (BQ returns datetime strings for
    the `cast` idiom — one more reason the cast has to go).
- Consequence: the current report SQL (e.g. `SHARED_API_REPORT_SQL.totalRequests`,
  `cast(timestamp_trunc(t.timestamp, hour) as datetime)` + named-alias group by)
  **fails as-is** through the live route (verified 500). The logs-previewer default
  query shape works as-is.
- A `usage.api-counts` substitute via `logs.all` sandbox SQL was **live-verified**:
  per-hour `countif(regexp_contains(r.path, '^/rest/'))`-style bucketing with
  `group by 1` returns the correct shape and correct merges.

Probes:

- edge-fn: `GET {kong}/functions/v1/` → **400 `{"msg":"missing function name in
request"}`** — a response generated by the Deno main worker actually executing
  code (stronger signal than realtime's 403-from-kong). A _named_ nonexistent
  function → 500 `InvalidWorkerCreation`, so the probe must use the bare path.
  `VERIFY_JWT=true` stacks: valid anon key → the same 400; missing key → 401 from
  worker code — all <500, all liveness proof.
- realtime: Logflare offers no stronger health signal than the websocket liveness
  check (logs reflect activity, not health; an idle stack logs nothing).

Degradation:

- Logflare container stopped → Studio route **500 `fetch failed` in ~148 ms**
  (connection refused fails fast). The unbounded-hang risk is code-level: no
  server-side Logflare fetch carries a timeout today (`lib/api/self-hosted/logs.ts:104`,
  both log-drain routes). Strict no-fallback re-verified live (proj-b NULL → 404
  `Analytics is not configured for this project`).

shared-db:

- Log streams are **stack-scoped**: vector routes by container name and tags a single
  `project`; kong/auth/storage events carry no database dimension. Same-stack projects
  cannot have their logs separated.

## §4 Server side — named-endpoint substitution (D3, D7)

`pages/api/platform/projects/[ref]/analytics/endpoints/[name].ts` gains a
substitution table for exactly four names: `usage.api-counts`, `service-health`,
`auth.metrics`, `functions.combined-stats`. For these, instead of forwarding the
name verbatim, the route sends an equivalent **`logs.all` sandbox SQL** (per-name
template, caller params like `interval` mapped in) and reshapes the rows to the
shape the existing frontend hook expects (pinned from the consumer types:
`project-log-stats-query`, `service-health-query`, `auth-overview-query`,
`functions-combined-stats-query`).

- Substitute SQL is written **dual-dialect** (ordinal group by, `countif`,
  `regexp_contains`, no datetime casts — all equally valid BigQuery), so no
  backend detection is needed and a BigQuery-backed self-hosted Logflare keeps
  working through the same rewrite.
- These Next API routes exist only in self-hosted deployments (cloud Studio never
  routes through them), so cloud is untouched **by construction**; the plain
  self-hosted behavior change (broken → working) is the sanctioned non-cloud fix.
- `logs.all` / `logs.all.otel` and every other name keep verbatim forwarding
  (unknown names keep today's error surface: Logflare 401 → 500 shape).
- The substitution lives in a new helper module (e.g.
  `lib/api/self-hosted/analytics-substitutes.ts`) so `[name].ts` stays a router.

Timeout (D6): `retrieveAnalyticsData` (`lib/api/self-hosted/logs.ts`) and every
Logflare fetch in both log-drain route files (`log-drains.ts`, `log-drains/[uuid].ts`)
gain `AbortSignal.timeout(15_000)` (constant, one place per module). Timeout/network
failure → existing `{error:{message}}` / 500 mapping.

## §5 Frontend — PG-compatible report SQL variants (D7)

The BQ-dialect report SQL definitions gain PG-compatible variants selected by a
picker gated on `!IS_PLATFORM` (`pickLogsQueryBuilder` precedent — cloud keeps the
BQ text byte-identical):

- `SHARED_API_REPORT_SQL` (SharedAPIReport — api-overview / postgrest / auth /
  realtime tabs' shared block, storage report),
- `data/reports/v2/auth.config.ts`, `data/reports/v2/edge-functions.config.ts`,
- `PRESET_CONFIG` / `queriesFactory` presets consumed by `useApiReport` /
  `useStorageReport`,
- the logs-chart count queries in `useLogsPreview` if they carry the broken idioms
  (plan recon item), `edge-functions-last-hour-stats`, unified-logs BQ builder
  (plan recon item — the `otelUnifiedLogs` flag is off in this deployment, BQ
  builder is the live path).

Variant rules (from §2): no `cast(as datetime)`; `group by`/`order by` ordinals
only (the shadowing-alias trap gets a pinned regression test); timestamps arrive
as epoch **microseconds** — the chart data layer must parse micros (logs-previewer
precedent; exactly where to normalize is a plan recon item, biased toward the
query-hook layer so chart components stay untouched).

## §10 Error handling summary

| Case                                              | Behavior                                                                      |
| ------------------------------------------------- | ----------------------------------------------------------------------------- |
| logflare fields NULL                              | 404 `Analytics is not configured for this project` (unchanged, live-verified) |
| Logflare down (connection refused)                | fast 500 `{error:{message:'fetch failed'}}` (~150 ms, live-verified)          |
| Logflare hung                                     | 15 s `AbortSignal.timeout` → same 500 error shape (new)                       |
| Substituted endpoint, empty tables                | 200 with empty rows → charts render their empty state, not an error           |
| Unknown endpoint name (not in substitution table) | verbatim forward → Logflare 401 → 500 shape (unchanged)                       |
| edge_function not deployed (non-catch-all kong)   | DISABLED (no-route marker); stock catch-all kong → <500 → ACTIVE_HEALTHY      |
| edge runtime down                                 | kong 502/503 → UNHEALTHY                                                      |
| ghost ref / zero-role                             | 404 / 403 via existing guards (unchanged)                                     |

## §12 Risks and accepted limitations

1. Logflare's BQ→PG translation layer is undocumented and version-pinned
   (1.43.1) — the variant rules in §2 are empirically bisected, not contractual.
   A Logflare upgrade could shift them; the pinned unit tests + README probe-path
   docs make drift diagnosable. Fail direction is a chart error state, not wrong
   data — except the shadowing-alias trap, which is why ordinals are mandatory and
   regression-pinned.
2. Substituted `usage.api-counts` counts kong request paths (`/rest/`, `/auth/`,
   `/storage/`, `/realtime/` prefixes) — semantics match "API requests through the
   gateway", which is what the stock stack can observe. Requests bypassing kong are
   invisible (same limitation as upstream's vector pipeline).
3. Per-service ingestion depends on each service's stdout verbosity (postgREST logs
   nothing per-request by default) — tabs over quiet services render empty, honestly.
   README notes this.
4. edge_function liveness+ can't distinguish "worker healthy" from "worker up but
   user functions broken" (same class as realtime's accepted weakness; honest beats
   fake-green). Stock catch-all kong reports ACTIVE_HEALTHY via 401-from-worker
   even if functions are undeployed-but-routed — accepted, same fail direction as
   M6.0's DISABLED caveat.
5. 15 s timeout may abort very large time-window queries on slow machines — surfaced
   as a chart error, retryable by narrowing the window; constant is one place.
6. Log retention/volume management for the PG backend is an operator concern
   (out of scope; README pointer only).
