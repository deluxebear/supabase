# M0.2 Patroni + pgBackRest PITR Spike

## Result

PASS on 2026-07-12 using an isolated Docker network with PostgreSQL 17,
Patroni 4.1.0, etcd 3.5.21, pgBackRest, one primary, and two synchronous
standbys.

## Scope

The destructive spike is implemented in
`apps/backup-operator/spikes/patroni-pitr.sh`. It creates uniquely prefixed
containers, networks, images, and volumes and removes only those resources on
exit. It accepts no caller-controlled SQL, paths, or resource names.

The spike verifies the following recovery contract:

1. Patroni forms a one-primary/two-standby cluster with synchronous mode
   enabled and both replicas streaming.
2. pgBackRest creates and checks a physical full backup and archives WAL.
3. Patroni is paused before restore coordination begins.
4. A manual switchover remains possible while Patroni is paused. Pause is
   therefore coordination state, not a fencing boundary.
5. During a short etcd outage, the current primary continues serving SQL, but
   cluster coordination is unavailable.
6. Every PostgreSQL process is stopped, the current primary is restored to a
   named restore point, and the two former replicas are erased.
7. Patroni resumes and automatically rebuilds both replicas from the restored
   primary on the new timeline.
8. Data written after the restore point is absent after recovery.

## Reproduction

From `apps/backup-operator`:

```bash
./spikes/patroni-pitr.sh
```

Final observed output:

```text
initial_leader=node1
post_switchover_leader=node2
pause_state=true
paused_switchover_http=200
dcs_outage_primary_sql=1
synchronous_standby_names=2 (node2,node3)
target_after_pitr=before-target
rebuilt_replicas=2
restored_timeline=3
RESULT=PASS
```

## Architecture consequences

- The Backup Operator must implement fencing independently of Patroni pause.
- Restore must stop all database nodes before mutating the primary data
  directory.
- Pre-restore replicas must not rejoin with their old data. They must be wiped
  and rebuilt from the restored primary.
- Docker deployments should mount the PostgreSQL parent directory and use a
  `PGDATA` child that is not the image-declared volume path. Patroni needs to
  rename and recreate the data directory during replica rebuilds.
- A DCS outage may leave PostgreSQL serving traffic while preventing safe
  coordination. The operator must fail closed instead of treating DCS loss as
  successful fencing.

## Limitations

This M0 spike proves the recovery mechanics and failure semantics. It does not
yet implement the production control API, durable workflow engine, leases,
Kubernetes adapters, object-storage repositories, or encrypted credentials.
Those are covered by subsequent milestones.
