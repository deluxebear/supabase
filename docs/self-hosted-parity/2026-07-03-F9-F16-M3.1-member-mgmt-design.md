## 3. 硬约束（M3.0 终审移交 7 条，逐条落位）

### 3.1 Shared-stack JWT secret 引用与角色优先级语义（约束 1）

引用 `docker/volumes/platform/README.md` "Shared-stack JWT secret" 段（README:564）：同栈注册的项目共用 JWT 签名密钥，凭据可见 ⇒ 同栈全部兄弟项目的数据面钥匙——M3.0 据此把 `secrets:Read` 收紧到 Owner/Administrator（README:659 起的 "M3.0: Roles and RBAC" 段）。本段成员端点不触碰凭据类路由，但**角色指派直接决定谁能过那道闸**，故显式声明：

**project_scoped 派生角色是 ADDITIVE GRANTS（加权），不是限制。** org 级 Admin 不会被某项目的 Read-only 派生角色"降级"——`effectiveBaseRoleName` 按 `BASE_ROLE_ORDER` 强度取最高（`rbac/expand.ts:38`），评估器对多角色取并集，设计如此（与云端语义一致，M3.0 Task 12 备注 + 终审确认）。TeamSettings 的 UI 呈现（一个成员同时有 org 角色徽标和项目 HoverCard）也基于该语义。README 在 M3.1 段补一句面向运维的表述。

## 5. 数据层设计（`lib/api/self-platform/`）

### 5.1 members.ts 扩展

- `listMembers(orgId)` → `Member[]` 形状的行：`organization_members` join `profiles`，`mfa_enabled = exists(select 1 from auth.mfa_factors f where f.user_id = pr.gotrue_id and f.status = 'verified')`（决策 2；GoTrue 自管 `auth` schema 在同一 platform 库），`role_ids` 聚合**仅限该 org 的角色**（`roles.organization_id = orgId` 过滤后 `array_agg`），全参数化。
- `getMemberInOrg(orgId, gotrueId)`：目标成员定位（不存在 → 路由 404）。
- Member 契约必填字段映射（platform.d.ts:8456）：`gotrue_id`/`username`/`primary_email` 取 profiles 列（01-schema.sql 三者均 not null）；`mfa_enabled` 如上；`role_ids` 如上；`metadata: {}`（必填、形状 `{[k]:unknown}`，空对象合法）；`is_sso_user: false`。
- `isOrgScopedRole(role: MemberRole): boolean = role.id === role.baseRoleId` 共享判据导出（§8 四处防呆共用）。

### 5.2 roles.ts（新文件）

- `listRolesV2(orgId)` → `OrganizationRoleResponse`（platform.d.ts:8690，**单数名**——`OrganizationRolesResponseV2` 不存在）：
  - `org_scoped_roles`：`base_role_id = id` 的 4 个基础角色，`projects: []`。
  - `project_scoped_roles`：派生角色（`base_role_id ≠ id`），`projects` 从 `role_projects` join `projects` 取 `{name, ref}`。
  - 每项 `{id, base_role_id, name, description|null, projects}`。

### 5.3 派生角色 CRUD（roles.ts）

- `createDerivedRole(orgId, baseRoleId, projectRefs)`：**refs 预校验独立查询先行**（`select id, ref from platform.projects where organization_id = $1 and ref = any($2)`，数量不符 → 400 `{message}`；独立于插入语句，避开 M1 I1-BUG 的同语句快照陷阱），然后单条多 CTE 原子插入 `roles`（`name = base.name || '-scoped-' || gen_random_uuid()`，满足 `unique(organization_id, name)`；UI 经 `base_role_id` 映射显示，name 不呈现）+ `role_projects`。
- `replaceRoleProjects(roleId, projectRefs)`（PUT 用）：同样预校验 + 原子 delete/insert `role_projects`。**空 `projectRefs` → 400，永不落库**（约束 2 HARD）。
- `unassignRoleWithGc(profileId, roleId)`：单条多 CTE——delete `member_roles` 行 + GC `delete from roles r where r.id = $2 and r.base_role_id <> r.id and not exists (select 1 from member_roles mr where mr.role_id = r.id and mr.profile_id <> $1)`。**快照安全性依赖 "其他持有者" 判式**（`profile_id <> $1`，对删除前快照成立）——不得写成裸 `not exists`（同语句快照会看见刚删的行，GC 永不触发；M1 I1-BUG 教训）。
- `removeMemberWithGc(orgId, profileId)`（DELETE member 用）：delete 该成员在该 org 的 `member_roles`（returning role_id）+ delete `organization_members` 行 + 同款 other-holder GC。
- `assignRole(profileId, roleId)`：`insert ... on conflict do nothing`（重复指派幂等）。

## 7. RBAC 细则

### 7.1 org 上下文（约束 4 的最小落法）

`PermissionCheckInput`（enforce.ts:15）加可选 `orgSlug?: string`；`checkPermissionWithContext` 传入时用之替代 `ctx.roles[0].orgSlug` 作 `doPermissionsCheck` 的 organizationSlug 实参。成员端点全部传路径 slug。不改现有 [ref] 调用点（默认行为不变，现有测试零改动）。

### 7.2 端点前置校验次序（404 先于 403，全端点统一）

1. slug 解析 org（`getOrganizationBySlug`）——未知 → 404 `{message:'Organization not found'}`（M3.0 org 详情先例）。
2. 调用者成员资格——非成员 → 404（同上，信息隐藏先例）。
3. `checkPermission(claims, {action, resource, orgSlug, data?})` —— 拒 → 403 `{message:'Forbidden'}`。**变更类必传 `data: {resource:{role_id}}`**（约束 3）。
4. 业务校验（目标成员存在 404、role_id 有效 400/404、`role_scoped_projects` 非空 400 等）。

### 7.3 读闸

GET members / roles / invitations / mfa-enforcement：`read:Read` on `organizations`（READ_ACTIONS 四基础角色皆有 → 任意持角色成员可读；零角色成员 → 403，沿 M3.0 决策 3 fail-closed；org shell 详情仍可见，与 M3.0 PASS 2 行为一致）。

### 7.4 Owner 保护的精确边界（documented semantics）

矩阵 deny 以 `role_id == 1`（`OWNER_ROLE_ID`）为条件：

- PATCH `role_id=1`（org-wide **或** 带 projects 的派生 Owner 创建）→ Admin 均被 deny（body role_id 恒为基础角色 id，recon §4）。✓
- DELETE role 路径 role_id 为**实际持有角色 id**：org-wide Owner（=1）受保护；**派生 Owner 角色（id≠1）Admin 可撤**——与 UI `rolesRemovable`（TeamSettings.utils.ts:34-42 用实际 role.id 评估）逐字一致，服务端不比 UI 更严也不更松。记 README 一句。
- DELETE member：对目标每个持有 role_id 逐一 `checkPermission(DELETE, 'auth.subject_roles', {resource:{role_id}})`，任一拒 → 403（镜像 MemberActions.tsx:67 `every` 语义）；零角色目标 → 一次不带 condition data 的基线检查（Admin/Owner 过，Developer/Read-only 拒——矩阵无此授权）。
- **last-Owner 防呆**（服务端新增，UI 侧 `hasMultipleOwners` 同意图）：DELETE member role 撤 org-scoped Owner、或 DELETE member 移除持 org-scoped Owner 的成员时，若该 org 无**其他** org-scoped Owner 持有者 → 400 `{message:'Cannot remove the last Owner of the organization'}`。防止组织锁死。

## 8. I1 四处防呆（纵深防御）

§5.3 已让空派生角色不可再产生；四个消费点仍统一换用 `isOrgScopedRole(role)`（`role.id === role.baseRoleId`）判据，使**存量/手误的空派生角色不授任何东西**：

| 位置                                              | 现判据（fail-open）                        | 改后                                                     |
| ------------------------------------------------- | ------------------------------------------ | -------------------------------------------------------- |
| `rbac/expand.ts:11` `expandPermissions`           | 空 `projectRefs` 展开为 org 级授权         | 派生且 `projectRefs` 空 → **skip**（零授权）             |
| `rbac/expand.ts:38` `effectiveBaseRoleName`       | `projectRefs.length === 0` ⇒ 适用一切 ref  | `isOrgScopedRole(role) \|\| projectRefs.includes(ref)`   |
| `list-user-projects.ts:127` `visibleProjectScope` | 空 `projectRefs` ⇒ `'all'`                 | `some(isOrgScopedRole)` ⇒ `'all'`；派生只贡献 projectIds |
| `organizations/index.ts:35` is_owner              | `baseRoleName==='Owner' && projectRefs 空` | `baseRoleName==='Owner' && isOrgScopedRole(role)`        |

各点配"空派生角色注入 → 零授权"回归测试；README M3.1 段补运维警告。
