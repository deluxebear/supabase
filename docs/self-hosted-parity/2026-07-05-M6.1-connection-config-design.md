## §1 Problem and goal

Since M2 a registry row is write-once: created by the register CLI or (since M5.0) the
create form, then editable only via manual SQL on the platform-db. The M2 README records
this as a known limitation ("no update/rotate-secret command"). A management plane whose
managed objects cannot be re-pointed, renamed, or key-rotated from the UI is incomplete —
and M6.2 (Logflare pipeline) needs a way to register per-project
`logflare_url`/`logflare_token` in the first place.

M6.1 delivers:

1. `PATCH /api/platform/projects/[ref]` — partial update of name, connection fields
   (db host/port/name/user/readonly-user/password, kong_url, rest_url, anon/service/jwt
   and optional publishable/secret keys), and logflare url/token.
2. A connection-config edit panel in Settings → General (self-platform mode), with
   write-only masked secret fields, explicit clear buttons for nullable fields, a
   probe-before-save gate, and shared-db/host-propagation awareness.
3. The upstream rename form on the same page starts working for free (same route+method).
4. The M2 "no update/rotate-secret" known limitation gets a "Superseded in M6.1" scope
   note (M6.0 T4 precedent for stale bullets).

## §3 PATCH contract

`PATCH /api/platform/projects/[ref]` (self-platform only; plain mode → 404
`Not available on this deployment`; `Allow` becomes `['GET','PATCH','DELETE']`
self-platform, `['GET']` plain).

```ts
{
  name?: string                          // ≤64 chars, non-empty when present
  connection?: {                         // refused 400 by name on shared-db rows (D1)
    dbHost?: string, dbPort?: number, dbName?: string,
    dbUser?: string, dbUserReadonly?: string,
    dbPass?: string,                     // required-secret semantics
    kongUrl?: string, restUrl?: string,
    anonKey?: string, serviceKey?: string, jwtSecret?: string,   // required-secret
    publishableKey?: string | null, secretKey?: string | null,   // nullable-secret
  }
  logflare?: { url?: string | null, token?: string | null }      // nullable (D5)
}
```

- **Deliberate divergence from the M5.0 POST body**: attach nests logflare inside
  `connection`; PATCH splits it out because D1's editability boundary (shared-db locks
  `connection`, allows logflare) must fall on a block boundary to gate cleanly.
- Field semantics: non-secret strings — absent = keep, present = validate + set, `''` →
  400 (not a mask; an empty host/url is never valid). Required secrets — `''`/absent =
  keep, new value = re-encrypt, `null` → 400. Nullable fields — `''`/absent = keep,
  new value = set (token re-encrypted), `null` = clear to NULL.
- `ref`/`stack_kind`/`stack_meta` anywhere in the body → 400 naming the field (D2).
  **All other unknown keys are silently ignored** (auth-config whitelist precedent —
  the upstream rename form may send cloud-only fields; rejecting them would break it).
- Empty effective patch (no editable field present) → 400.
- Check order (house pattern, M5.0 DELETE precedent): 405 skeleton → array-ref 400 →
  `guardProjectRoute(PermissionAction.UPDATE, resource 'projects')` (ghost → 404 before
  403, MFA chokepoint included) → body validation 400s → shared-db gate 400 → probe →
  persist → propagate → cache clear → 200.
- Response 200: same shape as the self-platform GET detail (`toProjectDetailResponse` of
  the fresh row — upstream rename mutation compatible) + additive
  `propagated_children: string[]` (empty when no propagation ran).

Validation rules for connection fields reuse the attach form's zod/parse rules
(`REF_PATTERN` class of constants; port integer range; URL shape for
kong/rest/logflare URLs) — one rule set, both write paths.

## §4 Data layer — `updateProjectConnection(ref, patch)` (projects-admin.ts)

1. Load the current row (`getProjectByRef`); unknown ref throws (route guard already
   404s ghosts before this runs — defense in depth, not a user path).
2. **Probe (D3)**: if `connection` present with ≥1 field, build the merged candidate
   `{dbHost, dbPort, dbName, dbUser, dbPass}` — new values where provided, else current
   row values, `dbPass` decrypted from `db_pass_enc` when not being replaced — and run
   the existing `probeConnection`. `{ok:false}` → throw `ProbeFailed` (route maps 400).
   Presence-based, not value-diff-based: a no-op connection patch still probes (harmless,
   and it re-confirms the stack is alive before touching ciphertexts).
3. **UPDATE**: SET clause assembled from a fixed column-whitelist map (column names come
   from a constant, values fully parameterized — same injection barrier class as M5.0);
   secrets pass through `encryptSecret` (fail-closed 500 if `PLATFORM_ENCRYPTION_KEY`
   missing, zero writes); `updated_at = now()` always included.
4. **Propagation (D7)**: if the row is `external` and the patch touched any cloned field
   (`db_host`, `db_port`, `db_user`, `db_user_readonly`, `kong_url`, `rest_url`, any of
   the 6 credential ciphertexts — **never** `db_name`/`name`/logflare), run a second
   sequential statement that **re-syncs the full cloned-field set** — all of the columns
   above copied from the host row's post-update values, not just the fields this patch
   touched:
   `update platform.projects set <all cloned columns from host values>, updated_at =
now() where stack_kind = 'shared-db' and stack_meta->>'host_ref' = $1 returning ref`.
   Full re-sync makes children exact clones again (also healing any historical drift
   from manual SQL) and is maximally idempotent. Two sequential statements, no
   transaction wrapper (M5.0 delete+GC precedent on this channel). Crash window between
   them leaves children stale; re-sending the same PATCH is idempotent and heals (README
   records the window). The trigger is presence-based like the probe.
5. **Cache (D4)**: `clearHealthCache(ref?: string)` — optional arg, no-arg keeps
   `cache.clear()` (existing tests untouched). After a connection change: clear the
   patched ref and each ref returned by the propagation statement. name/logflare-only
   patches do not clear (probe results are independent of both).

Errors reuse the M5.0 error-class → HTTP mapping pattern (`ProbeFailed` → 400; new
`ImmutableField`/`SharedDbLocked`-class 400s may be plain route-level validation rather
than thrown classes — plan's choice, contract identical).

## §5 GET extension (same route, self-platform branch only)

The self-platform GET `[ref]` response gains an additive `self_platform` block:

```ts
self_platform: {
  stack_kind: string, host_ref: string | null,        // from stack_meta
  db_host: string, db_port: number, db_name: string,
  db_user: string, db_user_readonly: string,
  kong_url: string, rest_url: string,
  logflare_url: string | null,
  secrets_set: { db_pass: true, anon_key: true, service_key: true, jwt_secret: true,
                 publishable_key: boolean, secret_key: boolean, logflare_token: boolean },
  shared_children: string[],                          // external rows: refs of shared-db
}                                                     // rows pointing at this host
```

**No plaintext secret ever leaves the server** — `secrets_set` booleans only
(test-pinned). Plain/cloud GET responses are byte-identical to today.

## §6 RBAC

`guardProjectRoute` with `action: PermissionAction.UPDATE ('write:Update')`,
`resource: 'projects'` — the first consumer of this pair. Matrix unchanged: Owner and
Administrator pass (the `%` wildcard; Admin's restrictive deny covers only DELETE),
Developer/Read-only 403. Deliberately one notch wider than Owner-only deregister:
editing a connection ≈ creating one (Admin already holds `write:Create` with secrets),
and the matrix as ratified in M3.0/M5.0 already says so. MFA enforcement rides the
existing `checkPermissionWithContext` chokepoint — no new code.

## §9 Error handling summary

| Case                                               | Response                                                                                   |
| -------------------------------------------------- | ------------------------------------------------------------------------------------------ |
| plain-mode PATCH                                   | 404 `Not available on this deployment`                                                     |
| ghost ref                                          | 404 Project not found (resolver-first via guard)                                           |
| no `write:Update` (Developer/Read-only)            | 403                                                                                        |
| body contains ref/stack_kind/stack_meta            | 400 naming the immutable field                                                             |
| `null` on a required secret                        | 400 naming the field as not clearable                                                      |
| `connection` block on a shared-db row              | 400 "managed by the host stack"                                                            |
| empty effective patch                              | 400                                                                                        |
| probe failure                                      | 400 `Could not connect to database: <cause>`, zero writes                                  |
| `PLATFORM_ENCRYPTION_KEY` missing                  | 500 fail-closed, zero writes                                                               |
| propagation statement fails after host row written | 500 with explicit message; host update persisted, children stale; same PATCH re-sent heals |
| cache clear                                        | in-process, cannot fail outward; write path does not depend on it                          |

## §11 Risks and accepted limitations

1. Probe covers the db channel only — kong/key mistakes surface via M6.0 health dots
   within one TTL after save (D3, accepted; same boundary as M5.0 attach).
2. Propagation crash window (host written, children not) — idempotent re-PATCH heals;
   README records it (same class as M5.0's insert-first windows).
3. Upstream rename form rendering in self-platform mode is a plan-recon item with a
   cheap fallback either way (panel name field).
4. Probe DSN does not URL-encode exotic password characters — pre-existing platform-wide
   debt recorded in the M5.0 backlog; not touched here.
5. A same-value connection patch still probes and propagates (presence-based) — trivial
   extra work, simpler and more testable than value-diffing; accepted.
