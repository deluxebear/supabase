## §1 Problem and goal

`pages/api/platform/projects/[ref]/infra-monitoring.ts` is the last M1-era empty
stub on the project routes: it returns `{data:[], yAxisLimit:0, format:'%', total:0}`
with **no auth, no RBAC guard, no mode-split** (pre-M3 legacy). daily-stats has no
route at all; `projects-resource-warnings.ts` returns `[]`. Consequently the
Database observability tab (13 chart configs, all `provider:'infra-monitoring'`),
the realtime tab's infra half (9 attributes), and the home instance diagram's
CPU/Disk/RAM/conns rows (`useComputeMetrics`, 7 attributes) render empty on
self-platform. M6.3 makes class-c infra metrics real for any registered stack:

1. Stack side: extend the M6.2 logs overlay so vector exports host + service
   metrics on one Prometheus endpoint (operator runbook, zero new containers).
2. Studio side: a resident sampler turns endpoint scrapes + per-project SQL into
   a platform-db time series; the infra-monitoring route serves the upstream
   chart contract from it; resource-warnings derives real exhaustion flags.
3. Registry/panel/CLI learn the `metrics_url`/`metrics_token` pair.
4. Folds in the M6.2 backlog bundle (D7).

## §2 Recon facts (2026-07-06, three-agent recon; live probes on the running stack)

- Cloud's `supabase-grafana` scrapes `https://<ref>.supabase.co/customer/v1/privileged/metrics`
  — officially unavailable self-hosted; self-hosted kong has **no metrics route**
  (all 20 routes enumerated). No reusable cloud artifact; kong stays untouched.
- vector 0.53 (already in `docker-compose.logs.yml`) supports `host_metrics`
  source (host CPU per mode / memory breakdown / disk IO / filesystem / network;
  needs `/proc` + `/sys` mounts and `PROCFS_ROOT`/`SYSFS_ROOT`), `prometheus_scrape`
  source, and `prometheus_exporter` sink. Config-only; no new containers.
- realtime and supavisor already serve Prometheus `/metrics` on container port
  4000 (live-verified: Bearer JWT signed by the stack's `METRICS_JWT_SECRET`,
  which the stock compose sets from the stack JWT secret; `osmon_*` CPU/RAM plus
  `realtime_connections_global_connected` etc. observed). Not host-published, not
  kong-routed — vector reaches them on the compose network.
- The db image preloads `pg_stat_statements` (verified in the running container's
  effective config); connections/db-size/WAL are all reachable over the existing
  pg-meta channel per project row — including shared-db children, where they are
  genuinely per-database.
- Frontend contract: both chart pipelines (database tab's `useInfraMonitoringQueries`
  and v2 configs' `runInfraMonitoringQuery`) hit
  `GET /platform/projects/{ref}/infra-monitoring` with `attributes[]`,
  `startDate`, `endDate`, `interval` (`1m|5m|10m|30m|1h|1d`; frontend-only `2m`
  aggregates client-side from `1m`) and parse via `mapResponseToAnalyticsData`:
  multi shape = `data[].{period_start, values}` + `series[attr].{format,total,yAxisLimit}`.
  `period_start` is the hardcoded x-axis key. `yAxisLimit` is passed through but
  effectively unread by charts (config `YAxisProps` wins). `databaseIdentifier`
  is the cloud replica selector — ignored server-side here (single database).
- The full backend attribute union is 57 names (`packages/api-types` platform.d.ts);
  home's `useComputeMetrics` needs exactly `avg_cpu_usage, ram_usage,
disk_fs_used_system, disk_fs_used_wal, pg_database_size, disk_fs_size,
pg_stat_database_num_backends` + the separate max-connections query.
- `useResourceWarningsQuery` is `enabled: IS_PLATFORM` (self-platform builds set
  it true), shape = 9 fixed keys per project; consumers: project cards, global
  banner, ComputeBadge.
- daily-stats is billing/usage semantics (egress, MAU, request totals); its
  consumers are org Usage/Billing pages and custom report blocks. No route today;
  stays that way (D5).
- `infra-monitoring.ts` today: `apiWrapper` **without** `withAuth`, no guard, no
  mode-split — the RBAC backfill is mandatory in this milestone.
- Register CLI has no logflare pairing validation; the pair invariant is enforced
  read-side (`logs.ts` throws `AnalyticsNotConfigured` when either is missing).
  The metrics pair follows the same read-side philosophy (§5: absent → L2 skipped).

## §5 Sampler engine + adapter — `lib/api/self-platform/metrics.ts` (new)

- Resident singleton, `IS_SELF_PLATFORM` only. Started from the Next.js
  instrumentation hook if the app supports it, else lazy-init on the first
  self-platform API touch (plan-time recon picks; either way a single
  `setInterval`-style loop with overlap protection — a cycle that overruns skips
  the next tick rather than stacking).
- Cycle (default `METRICS_SAMPLE_INTERVAL_MS = 60_000`): list all registered
  rows once (status-independent — gating on `status` would stop sampling a row
  the probe engine marked UNHEALTHY and never resume unprompted), then per row
  in parallel with per-row isolation (failure → `console.warn`, other rows
  unaffected — probe-engine precedent). L1 statements are individually isolated
  too: a failing statement (e.g. `pg_ls_waldir` privilege on a locked-down
  external stack) drops only its own attributes that cycle:
  - **L1 (every row)** over the existing pg-meta channel:
    `pg_stat_activity` grouped by usename → `client_connections_postgres/
authenticator/supabase_admin/supabase_auth_admin/supabase_storage_admin/other`
    and `pg_stat_database_num_backends`; `pg_database_size(current_database())`;
    WAL footprint via `pg_ls_waldir()` sum → `disk_fs_used_wal`;
    `max_connections` → `max_db_connections`. Genuinely per-database on shared-db
    children.
  - **L2 (rows with `metricsUrl`)**: `GET metrics_url` with
    `AbortSignal.timeout(15_000)` (M6.2 D6 constant), `Authorization: Bearer
<metricsToken>` when set. Parse Prometheus text; adapter maps to attributes.
- Adapter = one static table `attribute → { source (prom name + labels | SQL),
kind (gauge | counter-rate | derived), format, yAxisLimit }`:
  - CPU: `host_cpu_seconds_total{mode}` deltas over the cycle →
    `cpu_usage_busy_system/user/iowait/irqs/other` (%, clamped 0-100),
    `avg_cpu_usage` = 100 − idle share, `max_cpu_usage` = same value at sample
    grain (single host; documented).
  - RAM: memory gauges → `ram_usage_total/used/free/cache_and_buffers/swap`
    (bytes) and `ram_usage` (%); `swap_usage` (%).
  - Network: receive/transmit counter deltas → bytes/s.
  - Disk IO: read/written bytes + ops counter deltas → `disk_bytes_read/written`,
    `disk_iops_read/write`.
  - Filesystem (root mount): `disk_fs_size` and `disk_fs_used` stored as-is;
    `disk_fs_used_system` = disk_fs_used − pg_database_size − wal (floored at 0) so the stacked disk chart's components stay self-consistent with the
    SQL-sourced series.
  - Service scrape: realtime series → `realtime_sum_connections_connected`,
    `realtime_connections_connected`, `realtime_channel_joins/events/
presence_events/db_events`, `realtime_payload_size`; supavisor tenant series →
    `supavisor_connections_active`. Exact prom names + which of the 9 realtime
    attributes are actually served = **T1 spike pins**; unattainable ones are
    simply absent from the table (honest empty series).
  - Counter math: first cycle after process start or a negative delta (service
    restart / counter reset) → skip rate attributes for that cycle (unit-pinned).
  - Not mapped (honest empty): `disk_io_budget`/`disk_io_consumption` (AWS burst),
    `ram_commit_*` (chart flag-hidden), replication-lag family (no replicas),
    pgbouncer family, `cpu_usage` plain and `cpu_usage_busy_idle` (charts do not
    request them), and everything else outside the union subset above.
- Samples are stored **post-computation** (final gauge/rate values) so the route
  is a pure aggregation; the sampler keeps the previous raw scrape in memory per
  ref for delta math.
- Retention: on-insert sweep `delete … where sampled_at < now() − interval '7 days'`,
  rate-limited to at most once per hour per process.
- Registry-facing writes go through `executePlatformQuery` with parameterized
  statements only (house injection rules).

## §6 Routes (D2, D5)

### infra-monitoring — stub → real

`pages/api/platform/projects/[ref]/infra-monitoring.ts`, standard mode-split
template (`databases-statuses.ts` sibling):

- plain branch: **byte-identical M1 stub** (no auth, same literal — zero-break).
- self-platform branch: param 400 validation (ref string, attributes present,
  valid interval enum) → **`guardProjectRoute` backfill** with
  `PermissionAction.ANALYTICS_READ` on `'projects'` (analytics `[name].ts`
  precedent; plan re-verifies the constant) → bucket query on
  `platform.metrics_samples` (`date_trunc` for `1m|1h|1d`, epoch-division
  bucketing for `5m|10m|30m`, avg per bucket, window from `startDate/endDate`)
  → assemble `InfraMonitoringMultiResponse`: `data[].{period_start (ISO),
values}` plus `series[attr] = {format, total, totalAverage, yAxisLimit}`
  (total = window sum for rate/count attributes, latest value for gauges —
  per-attribute flag in the adapter table; yAxisLimit from the table, unread by
  charts today).
- **Always 200 after the guard**: attributes without samples return empty series
  (unconfigured `metrics_url` → L2 attributes empty while L1 attributes flow).
  Deliberately different from logflare's strict `AnalyticsNotConfigured` 404 —
  L1 makes partial data the normal state; per-attribute honesty replaces the
  404 wall. `databaseIdentifier` is accepted and ignored.

### resource-warnings — `[]` → derived

`pages/api/platform/projects-resource-warnings.ts`:

- plain branch: byte-identical `[]`.
- self-platform branch (existing `withAuth` stays; list semantics): resolve the
  caller's visible projects (existing list-user-projects helper), then per
  project read the latest `avg_cpu_usage`, `ram_usage`, and derived disk-percent
  (`disk_fs_used / disk_fs_size`)
  samples within the last ~5 minutes: ≥90 → `'critical'`, ≥80 → `'warning'`,
  else `null` for `cpu_exhaustion`, `memory_and_swap_exhaustion`,
  `disk_space_exhaustion`. All other keys `null`; `is_readonly_mode_enabled:
false`; stale/no samples → all-null row (honest). One samples query batched
  across refs, not N round-trips.

## §10 Error handling summary

| Failure                                      | Behavior                                                                     |
| -------------------------------------------- | ---------------------------------------------------------------------------- |
| L2 scrape timeout/refused/non-200/bad token  | warn + skip L2 for that row that cycle; L1 still sampled and written         |
| L1 SQL failure (stack db down)               | warn + skip row's L1 that cycle (probe engine owns health signaling, not us) |
| First cycle / counter reset (negative delta) | rate attributes skipped that cycle (designed gap, unit-pinned)               |
| Platform-db write failure                    | warn, cycle continues with other rows; never throws out of the loop          |
| Route query failure                          | existing `{error:{message}}` 500 shape                                       |
| Unknown/unmapped attribute requested         | empty series in response (200)                                               |
| No samples in window                         | empty `data`, zeroed series metadata (200)                                   |
| Warnings: stale or missing samples           | all-null warning row for that project                                        |
| Sampler down (Studio restart/downtime)       | honest gaps in charts; no backfill                                           |

## §12 Risks and accepted limitations

- **VM-host semantics**: on Docker Desktop/OrbStack, `host_metrics` reads the
  Linux VM, not the Mac; on Linux prod hosts it is the real host. Runbook states
  it; accepted.
- **Stack-level host metrics**: one stack = one host series; multiple projects
  on one stack see the same host numbers (shared-db hint covers the child case).
- **Sampler is single-process**: two Studio instances would double-sample
  (harmless duplicates averaged by bucketing, but wasteful). Internal-platform
  assumption accepted; dedup/locking is backlog.
- **Gaps are honest**: Studio downtime, first-cycle skips, and scrape failures
  leave visible chart gaps; no interpolation.
- **`max_cpu_usage` equals `avg_cpu_usage`** at our single-host sample grain
  (upstream distinguishes multi-node percentiles); documented, not faked.
- **Realtime attribute coverage depends on what the service actually exports**
  (T1 spike pins; unexported series stay empty rather than approximated).
- **7-day retention / 60s grain** bounds table size (~10k rows/attribute/week
  per project); `1d` interval charts over long ranges show at most 7 points.
- Token-less exporter is plain HTTP on the operator's network; runbook flags
  that `metrics_token` exists for fronting proxies, not vector itself.
