# M0.1 ControlStore Recovery-Domain Spike

- Date: 2026-07-12
- Linear: IFN-13
- Result: PASS
- Harness: `apps/backup-operator/spikes/control-store-recovery-domain.sh`
- Reusable guard: `apps/backup-operator/internal/recoverydomain`

## Question

Can an independent PostgreSQL ControlStore and an external SQLite ControlStore remain authoritative while a managed PostgreSQL cluster is stopped and physically restored to an earlier point? Can registration reject a control store that shares the target's PostgreSQL system identifier or storage/lifecycle domain?

## Isolation

The harness creates resources with a per-process `codex-controlstore-spike-*` prefix:

- one target PostgreSQL container and data volume;
- one independent control PostgreSQL container and data volume;
- one target-local POSIX pgBackRest repository volume;
- one SQLite database under a temporary host directory;
- one empty init-script volume so the test does not execute repository or user-provided SQL.

The cleanup trap removes every prefixed container, volume, and temporary directory. All SQL is hardcoded in the harness. No caller input is accepted or interpolated into SQL, resource names, paths, or pgBackRest arguments.

## Procedure

1. Start target and independent control PostgreSQL clusters.
2. Record each cluster's `pg_control_system().system_identifier` and Docker volume identity.
3. Store a `running` job record in independent PostgreSQL and external SQLite.
4. Verify the Go guard accepts different system identifiers and data domains.
5. Verify the guard rejects the target system identifier reused as the control system identifier.
6. Verify the guard rejects the target data volume reused as the control data domain.
7. Configure a target-local pgBackRest POSIX repository and WAL archiving.
8. Create a full backup containing `before-target`.
9. Create a named restore point and then commit `after-target`.
10. Stop the target and execute a real pgBackRest physical PITR to the named restore point.
11. Start the restored target and verify only `before-target` remains.
12. Verify both independent control records still report `running`.

## Binding output

```text
control store is outside the managed recovery domain
target_system_id=7661643058634723362
control_system_id=7661643059306856482
target_after_pitr=before-target
postgres_control_after_pitr=running
sqlite_control_after_pitr=running
RESULT=PASS
```

The numeric system identifiers are ephemeral fixture values. The binding behavior is that the identifiers differ and both independent control records survive target PITR.

## Implementation decisions

### Fail closed on incomplete identity

Registration requires both:

- PostgreSQL system identifier;
- deployment-specific data/lifecycle domain identity.

Missing either observation returns `ErrMissingIdentity`. A system identifier check alone does not detect two different PostgreSQL clusters sharing a volume lifecycle. A volume check alone does not detect two logical databases inside one PostgreSQL cluster.

### Keep authoritative state outside the target

The all-in-one `_platform` database remains unsuitable as the authoritative backup job store when it is inside the target PostgreSQL system. It may receive a compatibility projection only.

Production must use independent PostgreSQL. Lightweight single-host mode may use SQLite only when the file is outside `PGDATA` and all database-managed volumes.

### Use the real pgBackRest binary in operator execution

The image's `/usr/bin/pgbackrest` is a safety wrapper that strips `--config` and switches to the `pgbackrest` user. The isolated spike therefore invokes `/usr/lib/pgbackrest/bin/pgbackrest.real` as the `postgres` OS user with a fixed, operator-owned config path. Production code must treat the executable path and execution identity as enrolled Agent configuration and must never accept either from an API request.

## Acceptance criteria

- [x] Independent PostgreSQL survives target stop and physical PITR.
- [x] External SQLite survives target stop and physical PITR.
- [x] Target data returns to the named restore point.
- [x] Same PostgreSQL system identifier is rejected.
- [x] Shared storage/lifecycle domain is rejected.
- [x] Missing identity fails closed in unit tests.
- [x] Harness cleans all isolated resources.

## Follow-up contract for M1

`ControlStore` registration must collect a provider-specific `DataDomain`:

- bare metal: canonical mount/filesystem identity outside `PGDATA`;
- Docker: volume ID or bind-mount filesystem identity;
- Kubernetes: PVC UID/storage lifecycle identity;
- independent remote PostgreSQL: its system identifier plus independently observed deployment/storage identity.

The Operator must refuse restore capability until this identity is complete and independent.
