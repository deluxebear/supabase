## §1 Motivation

M6.4 shipped container-granular infra metrics for **compose** stacks: the adapter
selects the project's Postgres container by the cAdvisor label `name="supabase-db"`.
On **k8s** that label is a containerd task hex (useless), and the metric topology
differs. D3 deferred the k8s branch until a real scrape existed; it now does. This
milestone adds the k8s branch to the existing dialect so a `stack_kind='k8s'` project
reports its Postgres **Pod's** CPU/RAM/network, reusing the compose compute path.

## §2 Binding fixture facts (load-bearing — none writable blind)

From the captured scrape of pod `supabase-db-0` (namespace `supabase`, container
`postgres`), standalone cAdvisor on containerd:

| Fact                                                                                                                     | Value                                                                                                          |
| ------------------------------------------------------------------------------------------------------------------------ | -------------------------------------------------------------------------------------------------------------- |
| identity labels                                                                                                          | `container_label_io_kubernetes_pod_namespace`, `…_pod_name`, `…_container_name`                                |
| `name=` label                                                                                                            | containerd task hex — **not** a stable identity; must NOT be matched on                                        |
| CPU / memory / spec series                                                                                               | on `container_label_io_kubernetes_container_name="postgres"`                                                   |
| **network series**                                                                                                       | on `container_label_io_kubernetes_container_name=""` (the pod sandbox / pause), interface `eth0`               |
| pod-sandbox noise                                                                                                        | a second `container_name=""` series exists per pod → **must be excluded** from CPU/mem                         |
| `machine_cpu_cores` / `machine_memory_bytes`                                                                             | present (8 / ~10.16 GB) → `%` denominators available                                                           |
| `container_spec_memory_limit_bytes`                                                                                      | equals the pod `resources.limits.memory` (1 GiB in the fixture) → the limit-denominator RAM% path is exercised |
| metric NAMES (`container_cpu_usage_seconds_total`, `container_memory_working_set_bytes`, `container_spec_cpu_period`, …) | **identical to the compose cAdvisor family** → compute math reused verbatim                                    |

## §3 Ratified decisions

- **D3.1 — The only k8s-specific logic is series SELECTION.** The metric names,
  denominators (`machine_*` / spec limit / cpu quota-period), counter-delta rate math,
  and the shared host-disk + realtime/supavisor service blocks are all reused. The
  compose-vs-k8s difference is confined to `LabelPred`s.
- **D3.2 — Generalize `computeContainerAttributes` to take a selector**, not a raw
  `containerName`:
  ```ts
  type ContainerSelector = { cpuMem: LabelPred; net: LabelPred; cpuDenom: LabelPred }
  ```
  `cpuMem`/`cpuDenom` select the workload container's cpu/mem/spec series; `net`
  selects the network series (already `interface !== 'lo'`). Compose and k8s each
  supply a selector; the arithmetic body is stack-agnostic.
- **D3.3 — k8s identity = two dedicated editable columns + reuse `container_name`.**
  `k8s_namespace` + `k8s_pod_selector` (new, migration `11-k8s-identity.sql`);
  `container_name` doubles as the k8s container leaf (`"postgres"`). NOT `stack_meta`
  — it is in `IMMUTABLE_FIELDS` (PATCH-rejected); R2 already moved compose identity out
  of `stack_meta` for exactly this reason. New columns mirror the proven M6.4
  `container_name` chain end-to-end (parser → set → CLI → GET block → panel) and add a
  `MISSING_K8S_COLUMN` degradation tier.
- **D3.4 — Pod match is exact `pod_name` equality.** `k8s_pod_selector` matches
  `container_label_io_kubernetes_pod_name` exactly (deterministic; matches the fixture
  and a StatefulSet primary). Prefix / label-based / multi-replica-aggregation /
  CNPG-primary-failover selection are explicit non-goals (future refinement).
- **D3.5 — Dialect choice lives in the sampler, keyed on the resolved row.** k8s
  identity present → k8s selector; else `container_name` present → compose selector;
  else host dialect (M6.3, byte-unchanged). `stack_kind` is threaded onto
  `ResolvedConnection` to disambiguate.
- **D3.6 — Zero route/frontend/`ATTRIBUTE_META` change.** Every k8s value maps to an
  existing attribute name; the source choice lives entirely in the sampler/adapter
  (M6.4's pre-committed flip-cost boundary).

## §4 Architecture

### §4.1 Adapter (`lib/api/self-platform/metrics.ts`)

- Refactor `computeContainerAttributes(prev, curr, containerName)` →
  `computeContainerAttributes(prev, curr, selector: ContainerSelector)`. Body is the
  current one with `mine`/`notLo`/`cpuDenominatorCores(...)` reading `selector.cpuMem`
  / `selector.net` / `selector.cpuDenom` instead of `l.name === containerName`.
- New `containerSelector(conn): ContainerSelector | null`:
  - **k8s** (`conn.stackKind === 'k8s'` and `k8sNamespace` + `containerName` set):
    - `podPred = (l) => l.container_label_io_kubernetes_pod_namespace === ns && l.container_label_io_kubernetes_pod_name === podSelector`
    - `cpuMem = cpuDenom = (l) => podPred(l) && l.container_label_io_kubernetes_container_name === container`
    - `net = (l) => podPred(l) && l.container_label_io_kubernetes_container_name === '' && l.interface !== 'lo'`
  - **compose** (`containerName` set, non-k8s):
    - `cpuMem = cpuDenom = (l) => l.name === containerName`
    - `net = (l) => l.name === containerName && l.interface !== 'lo'`
  - else `null` (→ host dialect).
- Sampler (current `metrics.ts:507`): `const sel = containerSelector(conn); const scraped = sel ? computeContainerAttributes(prev, curr, sel) : computeScrapeAttributes(prev, curr)`.
- `cpuDenominatorCores` reads `CONTAINER.specCpuQuota`/`specCpuPeriod` via `selector.cpuDenom`; `machine_*` are global (no pred), unchanged.

### §4.2 Data model

- `docker/volumes/platform/migrations/11-k8s-identity.sql`: `alter table platform.projects
add column if not exists k8s_namespace text, add column if not exists k8s_pod_selector text;`
  (idempotent; manual-apply on upgrade, like every prior migration).
- `projects.ts`: add `k8s_namespace`, `k8s_pod_selector` to `PROJECT_SELECT_COLUMNS`,
  the `ProjectRow` type, and the row mapper; add `MISSING_K8S_COLUMN =
'column "k8s_namespace" does not exist'` degradation tier ordered **before**
  `MISSING_CONTAINER_COLUMN` (newest column first, mirroring the existing tier ladder)
  so a pre-11 db degrades the two columns → null instead of 500. **Load-bearing (verified
  against the current ladder):** the column list is ordered oldest→newest, and Postgres
  reports the **leftmost** missing column (= the oldest column a given db vintage lacks),
  which is exactly its tier boundary. So the two k8s columns append **last** (after
  `container_name`), the `MISSING_K8S_COLUMN` check runs **first** (newest-first, before the
  `container_name` check), and its retry tier is the current `PROJECT_SELECT_COLUMNS`
  (renamed `M64_SELECT_COLUMNS`: has `container_name`, no k8s). The final row mapper adds
  `k8s_namespace`/`k8s_pod_selector` `?? null`.
- `resolve-connection.ts`: add `stackKind: string`, `k8sNamespace: string | null`,
  `k8sPodSelector: string | null` to `ResolvedConnection`; map from the row (registry
  path) and leave null in the env-fallback path (env fallback is the plain-self-hosted
  'default' = compose; k8s never uses it).

### §4.3 Registration / identity chain (mirror `container_name`)

- `projects-admin.ts` PATCH parser: accept a nested `k8s: { namespace, pod_selector }`
  object (or `null` to clear), validated as strings, → `value.k8sNamespace` /
  `value.k8sPodSelector`; add both to the "no editable fields" guard.
- Insert (register) + `updateProjectConnection` set-list: `set('k8s_namespace', …)`,
  `set('k8s_pod_selector', …)` (plain columns, no encryption/masking — identity is
  non-secret).
- GET `self_platform` block: surface `k8s_namespace` / `k8s_pod_selector` (plain values,
  like `container_name`).
- `docker/scripts/platform/register-project.ts`: `--k8s-namespace` / `--k8s-pod-selector`
  flags + env pickup, mirroring `--container`.
- Panel (`SelfPlatformConnectionPanel`): two `textField`s (namespace, pod selector),
  shown when `stack_kind === 'k8s'`; reuse the existing `container_name` field for the
  container leaf. i18n the labels.

### §4.4 Degradation / honest boundaries

- Network-on-sandbox is a k8s fact, mapped honestly (not a defect).
- Partial identity (namespace/pod set, container missing/wrong) → cpu/mem attrs simply
  absent that cycle; no 500 (statement/selector isolation already in the sampler).
- `pre-11` platform-db → `MISSING_K8S_COLUMN` tier → k8s columns null → host dialect
  fallback (no crash).
- Disk stays L1 SQL (D4); container-level iowait/irq absent (cgroup has none) — same as
  compose container mode.

## §5 Testing (fixture-bound)

`metrics.test.ts` gains a k8s-dialect group binding `__fixtures__/cadvisor-k8s-scrape.prom`.
Build `prev`/`curr` snapshots from the fixture (offset the counters for the delta) and
assert:

1. CPU% derived from `container_name="postgres"` cpu counters over machine cores.
2. `ram_usage` uses `container_spec_memory_limit_bytes` (1 GiB) as denominator (limit
   path), `ram_usage_used` from that container's working set.
3. `network_receive_bytes`/`network_transmit_bytes` come from the `container_name=""`
   sandbox series (non-zero), and the postgres-container network is NOT double-counted.
4. The pause sandbox's cpu/mem are **excluded** (a fixture with pause cpu ≠ postgres cpu
   proves the selector picks postgres, not the sandbox or a sum).
5. `containerSelector` returns the compose selector for a `container_name`-only row and
   `null` for a bare row (host fallback) — a pure unit test, no fixture.

Also: a `projects.ts` degradation test for `MISSING_K8S_COLUMN` (pre-11 db → null, not
throw), mirroring the existing `MISSING_CONTAINER_COLUMN` test.

## §6 Reuse / change boundary

- **Unchanged**: sampler loop + isolation, `platform.metrics_samples`, the
  `infra-monitoring` route + response contract, `ATTRIBUTE_META`, `database-charts.ts`,
  the entire L1 SQL half, `computeScrapeAttributes` (host), `computeHostDisk`,
  `computeServiceAttributes`, the compose container path (behaviourally — it now flows
  through the selector but produces identical output).
- **Changed**: `computeContainerAttributes` signature (selector) + the compose call
  site; new `containerSelector`; `+2` registry columns + migration + degradation tier +
  `ResolvedConnection` fields; parser/CLI/GET/panel k8s identity; k8s test group.

## §7 Non-goals

- k8s collection manifests (cAdvisor DaemonSet etc.) — already delivered in
  `docker/k8s/single-project/`.
- Prefix / label-selector / multi-replica-aggregation / CNPG-primary-failover pod
  selection (D3.4).
- kubelet `/metrics/cadvisor` or kube-prometheus source variants (this milestone binds
  the cAdvisor-DaemonSet fixture only; other sources are future dialects).
- Any route / frontend / attribute-schema change (D3.6).
