## 3. Data model — 05-invitations.sql

House rule "no tables ahead of use" satisfied: table lands in the milestone that uses it.
Idempotent, 04-roles.sql conventions (create table if not exists; README manual-upgrade paragraph).

```sql
create table if not exists platform.invitations (
  id bigint generated by default as identity primary key,
  organization_id bigint not null references platform.organizations(id) on delete cascade,
  invited_email text not null,            -- stored lowercased; create route normalizes
  role_id bigint not null references platform.roles(id),  -- base org-scoped role only (route-enforced)
  role_scoped_projects text[],            -- project refs, NULL = org-wide grant on accept
  require_sso boolean not null default false,  -- stored for contract fidelity; never enforced (no SSO)
  token uuid not null default gen_random_uuid() unique,   -- 122 bits entropy, non-enumerable
  invited_by bigint not null references platform.profiles(id),
  invited_at timestamptz not null default now(),
  expires_at timestamptz not null default (now() + interval '24 hours'),  -- MUST match UI isInviteExpired
  accepted_at timestamptz                 -- consumed marker; row retained for the 401 mapping (§5.3)
);

create unique index if not exists invitations_org_pending_email_uq
  on platform.invitations (organization_id, invited_email)
  where accepted_at is null;              -- one pending invite per email per org
```

Notes:

- `role_id` restricted to base roles at route level (same laundering guard as M3.1 PATCH:
  reject non-base ids); scoped grants happen at accept via derived-role machinery, mirroring
  PATCH member V2 semantics.
- `role_scoped_projects` stores refs (not project ids): refs re-validated at accept time so
  projects deleted between invite and accept fail loud, not silently mis-grant.
- Revoke = hard DELETE (token thereafter → `token_does_not_exist` → UI "invalid" screen).
  Accept = keep row, set `accepted_at` (enables the "already accepted" 401 mapping).
- Expired pending rows stay listed (UI shows Expired badge + Resend); no reaper needed at
  internal scale (README note).

## 4. Email architecture (D1/D2 provisional)

Create-invitation route, per email, all-or-nothing:

1. validate + normalize (lowercase, dedupe input, syntactic email check)
2. reject → `failed[]`: already a member (organization_members row exists for a profile with this
   primary_email in this org — matches UI `categorizeInviteEmails` semantics, which treats every
   listed member incl. zero-role as "already in org"; workaround for zero-role users is direct
   role assignment, README note), already pending (partial unique index), unknown role /
   non-base role / ghost refs (whole-request 400 for role/refs — they are request-level, not
   per-email, matching PATCH member V2 conventions)
3. insert invitation row
4. dispatch email (D1):
   - try GoTrue admin `POST /invite` `{email, data:{org_slug}, redirect_to: SITE_URL/join?token=<token>&slug=<slug>}`
     with `Authorization: Bearer <mintServiceJwt(PLATFORM_JWT_SECRET, 'service_role', 60)>`
   - on "already registered" class error → fallback `POST /otp` `{email, create_user:false,
redirect_to: <same join URL>}` (magiclink for existing account)
   - both send through platform-auth SMTP (Mailpit locally); `GOTRUE_URI_ALLOW_LIST`
     already covers `SITE_URL/**` so redirect_to is accepted
5. email dispatch failure → delete the row just created, `failed[]` entry
   `'failed to send invitation email'` (invariant: pending row ⇒ email was sent; without
   copy-link UI a row without email is dead weight). GoTrue-side user possibly left in invited
   state on partial failure — zero-role, fail-closed, harmless (spike confirms actual behavior).
6. row + email OK → `succeeded[]`.

Task 7 spike (live, before implementation locks): (a) `/invite` works under
`GOTRUE_DISABLE_SIGNUP=true`; (b) `/invite` on existing email → error shape for fallback
trigger; (c) `/otp` with `create_user:false` sends magiclink and does NOT create; (d) whether
mail-send failure rolls back GoTrue user creation. Findings recorded in the ledger; if (a)
fails (GoTrue version quirk), fallback design = admin `/admin/users` create + `/otp` — same
surface, decision recorded then.

Compose (D2): new `platform-mail` service (axllent/mailpit, SMTP :1025, HTTP UI/API :8025
host-mapped via `PLATFORM_MAILPIT_HOST_PORT`, default-on in docker-compose.platform.yml) +
platform-auth gains `GOTRUE_SMTP_HOST/PORT/USER/PASS/ADMIN_EMAIL/SENDER_NAME` env-driven from
docker/.env with Mailpit defaults; README documents real-SMTP override (user fills creds;
placeholders only in the repo — hard rule: no real credentials authored by Claude).

New user first-login path: invite email → GoTrue `/verify?type=invite` → session lands on
/join (fragment tokens, gotrue-js picks up) → boot creates profile + default-org zero-role
membership (M1 chain) → by-token GET → accept. Password is unset until the user sets one in
account settings (documented; recovery flow also available once SMTP exists).

## 5. Endpoint semantics

Shared skeleton (M3.1 house pattern): `IS_SELF_PLATFORM` 404 → 405 (Allow header, nested
`{data:null,error:{message}}` shape) → 400 param validation → guard → business. Errors top-level
`{ message }`. Siblings `*.self-hosted.test.ts` full-byte 404. Bracket-named route tests live in
`apps/studio/tests/pages/...` mirror (M3.1 Turbopack lesson).

### 5.1 GET …/members/invitations (make M3.1 stub real)

Guard: unchanged from the M3.1 stub — `guardOrgRoute` READ `organizations` (any member).
Rationale (recon-verified): GET members uses the same guard, and the two requests feed one
`Promise.all` in `organization-members-query.ts` — a stricter guard here would break the whole
members list for exactly the roles that can see it today. matrix.ts `user_invites` rows are
CREATE/DELETE (the restrictive Owner-deny) — listing is org-member visibility, not a
member-management mutation. Returns pending rows only (`accepted_at is null`), mapped to
`InvitationResponse` (id, invited_at ISO, invited_email, role_id).

### 5.2 POST …/members/invitations (create, batch)

Guard: `guardOrgRoute` CREATE `user_invites` + per-request `checkPermission` with
`{resource:{role_id}}` condition data (Admin-invites-Owner denied by DENY_OWNER_ROLE_GRANTS —
M3.1 PATCH precedent). Body: `emails[]` (fallback to deprecated `email` — normalize to array);
`role_id` must be base org-scoped role of this org (laundering guard); `role_scoped_projects`
refs → `Set` dedupe (M3.1 Minor#2 precedent) → `getOrgProjectIdsByRefs` validation → 400 with
miss list on ghosts. Then §4 per-email pipeline → 201 `CreateInvitationResponse`.

### 5.3 GET …/invitations/{token} (by-token, capability semantics)

Auth: session required (withAuth) — **no org-membership guard** (the invitee is by definition
not a member; possession of the token is the capability). Info-hiding: org slug + unknown token →
`{token_does_not_exist: true, organization_name: '', …}` 200 — never reveals whether the org
exists; token looked up scoped to the slug's org so a token leaked from another org doesn't
resolve here.

Response mapping (order matters):

1. row consumed (`accepted_at` not null) → **401 `{message: 'Failed to retrieve organization invitation'}`**
   (substring-matches utils:42 → "already accepted or declined" screen)
2. no row for (org, token) → 200 with `token_does_not_exist: true`
3. org `enforce_mfa` && `claims.aal !== 'aal2'` → **403 `{message: 'MFA required to join this organization'}`**
   (substring 'MFA required' → MFA setup screen; checked before revealing org_name/email_match)
4. else 200: `{authorized_user: true, email_match: lower(claims.email) === invited_email,
expired_token: expires_at < now(), invite_id: id, organization_name, sso_mismatch: false,
token_does_not_exist: false}`

### 5.4 POST …/invitations/{token} (accept)

Auth: session required, no membership guard. Server re-checks everything the GET showed
(fail-closed; the GET is advisory UI):

- consumed → 401 (same message as 5.3.1); missing → 404 `{message:'Invitation not found'}`
  (utils maps 404 → invalid); expired → 400 `{message:'Invitation has expired'}`;
  email mismatch → 403 `{message:'Invitation was issued for a different email'}`;
  enforce_mfa && aal!=='aal2' → 403 'MFA required…' (exact same literal).
- Then the write. **Ordering (snapshot rules per M1 I1-BUG ledger):**
  - org-wide grant path: single multi-CTE statement — claim
    (`update … set accepted_at = now() where token = $1 and accepted_at is null and expires_at > now()
returning …`) → ensure membership (`insert … organization_members … on conflict do nothing`,
    forced dependency on claim CTE) → `insert member_roles` gated **insert-select on the
    membership row** (Task 8 anchoring; select-from-inserted-CTE union existing-row form to
    avoid same-snapshot visibility trap — exact predicate form in the plan, regression-pinned
    like M3.1 Task 2).
  - scoped path (role_scoped_projects non-empty): refs → ids re-validation, then
    `createDerivedRoleWithAssignment` (reused; already snapshot-safe + non-empty-guarded)
    **first**, then the claim+membership CTE; if claim returns 0 rows (raced/expired between
    check and claim) → compensate `unassignRoleWithGc` and return the 401/400 per re-read state.
    Crash window between create and claim leaves a visible, admin-removable derived grant —
    accepted at internal scale, README-documented (same risk class as M3.1 accepted items).
- Response: 201 empty-object (accept-mutation ignores body; match cloud's 2xx).

### 5.5 DELETE …/invitations/{id} (revoke)

Guard: `guardOrgRoute` baseline DELETE `user_invites` (no condition data) → row lookup by
(org, id), 404 `{message:'Invitation not found'}` → per-target `checkPermission` DELETE
`user_invites` with `{resource:{role_id: row.role_id}}` (Admin can't revoke an Owner invite —
mirrors UI `canRevokeInvite` gating and M3.1 DELETE-member two-layer precedent) → hard delete →
200 `{}`. Pending-or-accepted: only pending rows are listed/addressable in UI; DELETE on an
accepted row 404s (id namespace still valid but semantically gone — simplest: lookup filters
`accepted_at is null`).

## 6. MFA enforcement (D3 provisional: join + session layer)

### 6.1 Join flow (hard requirement)

§5.3.3 + §5.4 emit the exact `'MFA required'`-containing message. Check = `claims.aal !== 'aal2'`
(typed claim). aal is chosen over factor-existence: fail-closed for enrolled-but-unchallenged
sessions; a user who enrolls mid-flow gets aal2 on challenge verify, then retries accept.

### 6.2 Session layer

`guardOrgRoute` and `guardProjectRoute` gain the check after membership/404 resolution and
before permission evaluation: org has `enforce_mfa` (column fetched alongside the existing org
lookup — no extra round trip) && `claims.aal !== 'aal2'` → 403
`{message: 'MFA required to access this organization'}`. Placement keeps 404-before-403 and
info-hiding intact (non-members still get 404, MFA state revealed only to actual members).

### 6.3 Accepted UX cost (recorded)

Our contract has no `organization_requires_mfa` field (M1 sanctioned adjustment) so there is no
dedicated frontend gate screen: existing members without MFA in an enforce_mfa org get 403
toasts on every org/project API until they enroll (account/security page works — it is
account-level, not org-guarded). Owner opt-in per org; README states this loudly. If D3 flips
to join-only, §6.2 is dropped and §6.3 disappears.

## 8. Hardening rider (D4): grants anchored to membership

`assignRoleToMember` (roles.ts:106) becomes insert-select gated on the
`organization_members` row of the target profile in the role's org (join through
`platform.roles.organization_id`); zero rows inserted when membership is gone → callers turn
that into 404/409 per route (PATCH member assign path). Closes M3.1 final-review #4
(PATCH-assign racing DELETE-member leaving an orphan grant that passes project-route guards).
`createDerivedRoleWithAssignment`'s member_roles insert gets the same anchoring predicate.
Accept flow (§5.4) is born anchored. Regression pins on the SQL predicate (M3.1 Task 2 style),
plus a live begin/rollback proof on platform-db (membership deleted → assign inserts 0 rows).

## 9. Security invariants (spec-hard, tests pin each)

1. Token capability: 122-bit random uuid, unique, never logged by our code; by-token responses
   never distinguish "org missing" from "token missing"; cross-org token does not resolve.
2. Fail-closed accept: every precondition re-checked server-side at POST time; claim is a single
   conditional UPDATE (atomic consume — no double-accept under concurrency).
3. Role laundering closed: create rejects non-base role_id; accept grants only what the
   invitation row says (role_id + refs re-validated at accept).
4. Condition data always passed on CREATE/DELETE user_invites guards (Admin/Owner boundary
   server-real, M3.1 lesson).
5. Info-hiding order preserved: 401 → 404 → 403 on guarded routes; MFA 403 only after
   membership established (session layer) or token possession proven (join flow).
6. No credentials in git: SMTP creds are docker/.env placeholders; PLATFORM_JWT_SECRET consumed
   from env, never printed.
7. Zero-break HARD: all new/changed routes have byte-asserting plain-mode siblings; compose
   changes touch only the platform stack.
8. SQL all `$n`-parameterized through executePlatformQuery; multi-step writes follow the
   snapshot rules (no same-statement visibility assumptions; forced CTE dependencies).

## 13. Risks and accepted limitations

- Accept scoped-path crash window (derived grant before claim) — visible + admin-removable,
  internal scale (§5.4). Mirrors M3.1 accepted-risk class.
- GoTrue user pre-creation on invite (D1): revoked invitation leaves a zero-role GoTrue account
  (fail-closed everywhere; RBAC proven M3.0/M3.1). Passwordless until self-set.
- Invited-then-never-accepts: pending row expires (24h), resend = delete+recreate; GoTrue-side
  invite token expiry is GoTrue-config (README note; magiclink resend covers).
- Zero-role legacy users can't be re-invited via UI ("already in org") — direct role assignment
  is the path (README).
- Session-layer MFA has no dedicated UI gate (§6.3) — toast-level UX, Owner opt-in.
- `email` claim staleness: claims.email is from JWT issuance; email changes mid-session could
  mismatch until refresh — GoTrue rotates JWT on refresh; accepted.
- Single-org assumptions persist (multi-org backlog: OWNER_ROLE_ID literal, org-owning-ref).
