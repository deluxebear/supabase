## §1 Problem and goal

Since M1, `/v1/projects/{ref}/health` and `/platform/projects/{ref}/databases-statuses`
are contract-minimal stubs that report every service `ACTIVE_HEALTHY` unconditionally
(`pages/api/v1/projects/[ref]/health.ts:28`). The ServiceStatus dots on the project home
are therefore fake-green for every attached stack, and the registry `status` column is
whatever registration wrote (never observed). M6.0 replaces this with real per-stack
probing from the Studio server — any attached compose/k8s stack becomes observable with
zero stack-side deployment.

## §2 Recon facts (verified)

- Contract `V1ServiceHealthResponse` (`packages/api-types/types/api.d.ts:5362`):
  `{ name, healthy (deprecated bool), status, error?, info? }`; `info` variants carry
  GoTrue `{name:'GoTrue',version,description}` and realtime connection details.
- Frontend consumes `status` strings `ACTIVE_HEALTHY | UNHEALTHY | COMING_UP | DISABLED`
  (`components/interfaces/ProjectHome/ServiceStatus.tsx:46-70`); the query asks for
  `services=auth,realtime,rest,storage,db` (`data/service-status/service-status-query.ts:25`).
- Kong (docker/volumes/api/kong.yml): `/auth/v1/*` proxies to GoTrue with key-auth on the
  general route (health path `/auth/v1/health` requires an `apikey`); the registry holds
  each project's anon key. Kong's no-route response body contains `no Route matched`.
- db probing channel exists: the resolver's `pgConnEncrypted` + `x-connection-encrypted`
  header against `${PG_META_URL}/query` (same mechanism as M5.0's `probeConnection`).
- `databases-statuses.ts` already carries an RBAC guard (M3.0); only its status source is
  a stub. `health.ts` (v1) has `withAuth: true` but no per-project guard.
- Edge Functions sub-status fetches a HARDCODED Supabase cloud URL
  (`data/service-status/edge-functions-status-query.ts:12`).
- Registry: `platform.projects.status` exists (text, unconstrained); no probe timestamp
  column.

## §3 Probe engine — `apps/studio/lib/api/self-platform/health.ts` (new)

```ts
export type ProbeService = 'db' | 'auth' | 'rest' | 'storage' | 'realtime'
export interface ServiceProbeResult {
  name: ProbeService
  status: 'ACTIVE_HEALTHY' | 'UNHEALTHY' | 'DISABLED'
  healthy: boolean // deprecated contract field, derived: status === 'ACTIVE_HEALTHY'
  error?: string
  info?: unknown // GoTrue health JSON passthrough for auth
}
export async function probeStackHealth(ref: string): Promise<ServiceProbeResult[]>
```

- Resolves the connection once (`resolveProjectConnection(ref)` — ghost refs throw
  `ProjectNotFound` → apiWrapper 404).
- Probes all five services in parallel (`Promise.allSettled`), per-service timeout
  **5s** (`AbortSignal.timeout(5000)`).
  - **db**: POST `${PG_META_URL}/query` `{query:'select 1'}` with
    `x-connection-encrypted: conn.pgConnEncrypted` (rw DSN — the connection the
    dashboard actually depends on; ro-DSN misconfig must not fake project death).
  - **auth**: GET `${conn.supabaseUrl}/auth/v1/health`
  - **rest**: GET `${conn.supabaseUrl}/rest/v1/`
  - **storage**: GET `${conn.supabaseUrl}/storage/v1/status`
  - **realtime**: GET `${conn.supabaseUrl}/realtime/v1/api/health`
    (exact 200-path pinned by a live spike in the plan — fallback candidate
    `/realtime/v1/health`; spike also confirms the auth/storage paths on the pinned
    self-hosted service versions)
  - HTTP probes send `apikey: conn.anonKey` + `Authorization: Bearer ${conn.anonKey}`.
- **Status mapping** (single table, unit-tested):
  | Observation | status | error |
  |---|---|---|
  | 2xx (db: query 2xx) | ACTIVE_HEALTHY | — |
  | 404 whose body contains `no Route matched` (Kong: service not deployed) | DISABLED | — |
  | any other non-2xx | UNHEALTHY | `HTTP <code>` + body message if JSON |
  | timeout / network error | UNHEALTHY | error message text |
  Error strings never include credentials (messages only; DSNs never echoed).
- **auth info passthrough**: on 2xx, if the body parses as JSON it is attached as `info`.
- **TTL cache**: module-level `Map<ref, {at:number, results:ServiceProbeResult[]}>`,
  **20s**. All five services probed and cached as one unit; callers filter. Multiple
  Studio instances each keep their own cache (accepted; single-process is the norm).

## §4 Routes + write-through

**`pages/api/v1/projects/[ref]/health.ts`** — becomes mode-split:

- `!IS_SELF_PLATFORM`: current static always-healthy stub, **byte-identical** (zero-break
  sibling pins it).
- self-platform: 405 skeleton → array-ref 400 → `guardProjectRoute(read:Read,
resource 'projects')` (new — also buys ghost→404 info-hiding) → `probeStackHealth(ref)`
  → filter to the requested `services` → map to the contract shape (`healthy` +
  `status` + `error` + `info`) → 200.
- After a probe (cache miss only), **write-through**: overall = db result (D3);
  `update platform.projects set status=$2, last_health_at=now() where ref=$1` executed
  only when the computed status differs from the row's OR `last_health_at` is older than
  60s (bounds write volume). COMING_UP rows get overwritten by observed truth
  (a stuck creation row resolving to UNHEALTHY is honest and self-explaining).

**`pages/api/platform/projects/[ref]/databases-statuses.ts`** — keeps its existing RBAC
guard and single-primary shape; the `status` value now comes from the db probe (shares
the cache; no extra probing).

**List badges**: no frontend change — the V2 project lists already render `row.status`;
write-through makes them show the last-observed state.

## §8 Error handling summary

| Case                                 | Behavior                                                                        |
| ------------------------------------ | ------------------------------------------------------------------------------- |
| ghost ref                            | 404 Project not found (resolver throw via guard)                                |
| zero-role member                     | 403 (guard read:Read 'projects')                                                |
| stack fully down                     | all services UNHEALTHY with error text; project status → UNHEALTHY; badges flip |
| service not deployed (Kong no-route) | DISABLED (not UNHEALTHY — honest)                                               |
| probe exception mid-flight           | that service UNHEALTHY (allSettled), others unaffected                          |
| write-through failure                | logged `console.warn`, response still 200 (observation ≠ persistence)           |
| repeated dashboard polling           | 20s cache absorbs; at most 1 probe burst per ref per TTL                        |

## §10 Risks and accepted limitations

1. Realtime/storage health paths vary by service version — plan opens with a live spike
   against the running stack (M5.0 T2 spike precedent); mapping table keeps DISABLED
   for no-route so older stacks degrade honestly rather than red-flagging.
2. Custom Kong configs could route health paths differently → worst case DISABLED/
   UNHEALTHY with error text; README documents the probe paths so operators can verify.
3. Write-through makes `status` an observed value; anything treating it as a manual
   lifecycle field changes meaning (audited: nothing does — resolver ignores it, lists
   display it).
4. Multi-instance cache divergence (each Studio process probes independently) — accepted
   at internal scale.
5. Aux-service DISABLED vs UNHEALTHY hinges on Kong's no-route message text — pinned by
   spike; if a gateway variant words it differently the result degrades to UNHEALTHY
   (fail-noisy, not fail-green).
