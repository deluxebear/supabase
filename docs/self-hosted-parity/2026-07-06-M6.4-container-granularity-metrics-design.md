## Plan-time refinements (post-ratify, 2026-07-07 — supersede where noted)

Two forks surfaced during writing-plans recon and were user-decided; they refine ratified decisions:

- **R1 refines D4/D7 — % denominator is the MACHINE (docker-stats semantics), not "limit-only".** Container CPU% = `Δcontainer_cpu_usage_seconds_total / elapsed / machine_cpu_cores`; RAM% = `container_memory_working_set_bytes / machine_memory_bytes`; when a container limit IS set (cpu quota / `spec_memory_limit>0`), the limit is the denominator instead. Rationale: the spike proved `supabase-db` has no limit, so "limit-only %" (literal D4) would leave the CPU/RAM % charts + home tiles EMPTY in the common shared-stack case — the exact case this milestone exists for. `machine_cpu_cores`/`machine_memory_bytes` come from cAdvisor's own output (still "no extra collection sources"). Disk stays L1 per-db (D4 unchanged); absolute-byte memory series still emitted alongside %.
- **R2 supersedes D5/§4.3 — identity is a DEDICATED COLUMN `container_name`, not a `stack_meta` key.** Recon found `stack_meta` is in `IMMUTABLE_FIELDS` (PATCH rejects it by name), so a `stack_meta.container` key could not be edited from the panel. A dedicated `container_name` column (migration `10-container.sql` + a `MISSING_CONTAINER_COLUMN` degradation tier) mirrors `metrics_url` end-to-end (parser → `set('container_name',…)` → CLI `$25` → GET block → panel `textField`), giving PATCH- and panel-editability and reusing the proven M6.3 column pattern instead of new jsonb-merge code. Cost: one migration + one degradation tier (both established M6.3 patterns). k8s pod-selector identity (D3) will still ride in `stack_meta` when built.
- **D8 resolved — `ram_commit_*` is honestly ABSENT.** The M6.3 host fixture confirms vector `host_metrics` emits no committed-memory series, and cgroups/cAdvisor have no commitment concept. No mapping is added; the Memory-commitment chart stays hidden/empty as in M6.3.
- **T1 collector pinned:** cAdvisor `v0.49.1` runs lean with `--store_container_labels=false --whitelisted_container_labels=com.docker.compose.service,com.docker.compose.project --disable_metrics=sched,percpu,memory_numa,cpuLoad,diskIO,disk,tcp,advtcp,udp,app,process,hugetlb,perf_event,referenced_memory,cpu_topology,resctrl` → 7.4 MB raw scrape drops to ~0.9 MB, no `container_fs_*`/blkio. Binding fixture: `__fixtures__/cadvisor-scrape.prom` (supabase-db `container_*` + `machine_cpu_cores`/`machine_memory_bytes`).
- **Frontend/route unchanged:** every container-mode value maps to an EXISTING `ATTRIBUTE_META` attribute name; the container-vs-host source choice lives entirely in the sampler/adapter, so the route, the 57-attr generated union, and `database-charts.ts` need no change.

## §1 Motivation

M6.3 shipped **host-level, stack-scoped** infra metrics: vector `host_metrics`
reads the whole VM's `/proc`+`/sys`, so CPU/RAM/filesystem are one series per
stack. When multiple projects share a stack (shared-db children, or several
registered projects on one compose host) they all see the _same_ VM-global
numbers. Supabase Cloud avoids this by giving every project its own dedicated
EC2 instance, so "whole machine" == "this project" for free — a deployment
premise the management-plane does not have.

M6.4 drops the granularity from "machine" to **the project's Postgres
container** (a project == one Postgres container/Pod), so container-granular
resource metrics are per-project by construction without dedicated machines.
Container isolation is cgroup-based in both compose and k8s, so one collection
family (`container_*`) covers both. Target: the Database observability tab (the
same `database-charts.ts` config M6.3 already feeds) reports _this project's_
CPU/RAM rather than the VM's.

## §2 T1 spike findings — BINDING (live 2026-07-06, OrbStack, cgroup v2)

Two collectors were spiked live against the running `supabase-db`. **cAdvisor
was chosen** (D1 below); its scrape is the binding fixture source.

### §2.1 cAdvisor (chosen path)

cAdvisor `v0.49.1` (arm64) runs cleanly on this OrbStack / cgroup-v2 stack
(crio/mesos/podman/containerd factory-registration failures in its log are
benign; the docker factory registers). Scrape `:8080/metrics` → 200.

Binding series for the Postgres container (label `name="supabase-db"`):

| Metric                                                            | Type    | Use                                                                                 |
| ----------------------------------------------------------------- | ------- | ----------------------------------------------------------------------------------- |
| `container_cpu_usage_seconds_total`                               | counter | total CPU seconds (counter-delta rate, reuse M6.3 math)                             |
| `container_cpu_user_seconds_total`                                | counter | User segment                                                                        |
| `container_cpu_system_seconds_total`                              | counter | System segment                                                                      |
| `container_memory_working_set_bytes`                              | gauge   | RAM used (169.9 MB live; matches `docker stats` ~158–171 MiB)                       |
| `container_memory_usage_bytes` / `_rss` / `_cache`                | gauge   | memory breakdown                                                                    |
| `container_spec_memory_limit_bytes`                               | gauge   | **RAM% denominator when a limit is set** — `0` when unlimited (supabase-db reads 0) |
| `container_network_receive_bytes_total` / `_transmit_bytes_total` | counter | **per-container network (bonus)**                                                   |
| `container_oom_events_total`                                      | counter | OOM signal (optional)                                                               |

**Container identity labels** (the reason cAdvisor was chosen over vector
cgroups): `name="supabase-db"` (stable across recreate), plus
`container_label_com_docker_compose_service="db"` and `..._project="supabase"`.
Also `id="/docker/<64-hex>"` and `image=`. Fixture: `m6.4-T1-cadvisor-supabase-db.prom` (trimmed, 173 lines).

**Cardinality caveat (binding for plan):** an unconfigured cAdvisor scrape is
~7.4 MB (all containers × all metrics × 25+ `container_label_*` each). Must be
trimmed via cAdvisor flags + a vector filter (see §4.1).

### §2.2 vector cgroups collector (rejected path — recorded for completeness)

vector 0.53 `host_metrics` **does** have a working `cgroups` collector on cgroup
v2 (`vector validate` passes; live run emits `host_cgroup_cpu_{usage,user,system}_seconds_total`

- `host_cgroup_memory_current_bytes` + anon/file breakdown, tagged
  `cgroup="docker/<id>"`). Rejected because its label carries only the container
  **id** (changes on `--force-recreate` → stale registration), it emits **no
  io/blkio and no memory limit**, and its names are `host_cgroup_*` — **not** the
  same family as cAdvisor's `container_*`. Fixture kept: `m6.4-T1-cgroups-scrape.prom`.

### §2.3 Corrections to the exploratory draft (from the spike)

1. Draft §3.1 "same `container_*` naming family, shared dialect" is **false** for
   the two collectors — but moot now: cAdvisor's `container_*` is used for both
   compose and k8s, so the family genuinely is shared **within the chosen path**.
2. Draft §3.4 "disk IO/throughput container-granular" is **not honestly
   achievable**: cAdvisor `container_fs_*`/`container_blkio_*` measure the
   container **rootfs/overlay**, not the Postgres data volume — charting them as
   "project disk IO" would mislead. Disk IO stays host-level (additive) or empty.
3. RAM% denominator confirmed absent unless the operator sets a container memory
   limit (`container_spec_memory_limit_bytes > 0`).

## §3 Ratified decisions

- **D1 — Collector: cAdvisor overlay** (not vector cgroups). +1 container, but
  stable `name=` labels (no recreate-staleness), native memory-limit series, and
  per-container network. Accepts the footprint over the zero-container vector
  option.
- **D2 — Source strategy: additive / fallback.** Keep the M6.3 `host_*` source;
  add the cAdvisor `container_*` source. The adapter reads container-level when
  the project has a registered container identity, else falls back to host-level.
  Single-project stacks keep the richer host signal (5-segment CPU, disk IO,
  RAM%); shared/multi-project stacks get honest per-project container metrics.
- **D3 — k8s: compose first, k8s is an extension point.** Build the
  per-`stack_kind` dialect **structure**; do **not** implement or bind the k8s
  branch (no live k8s stack to capture a binding cAdvisor/kubelet fixture →
  fixture-is-binding forbids writing it blind). `stack_meta` reserves the k8s
  identity shape.
- **D4 — Disk & RAM% honest degradation, no extra collection sources.** RAM in
  absolute bytes; percentage only when `container_spec_memory_limit_bytes > 0`.
  Disk **usage** stays the L1 SQL per-db path (`pg_database_size` + WAL,
  unchanged). No container-rootfs disk-IO chart; no volume-partition collector.
- **D5 — Identity key** = `stack_meta.container` stores the container **name**
  (e.g. `supabase-db`; cAdvisor `name=` gives it directly, stable across
  recreate, portable). compose-service label is the secondary match.
- **D6 — Single endpoint** = cAdvisor is scraped **by vector** and re-exported on
  the existing `:9598`; the registry keeps one `metrics_url` per project. This is
  forced by D2 (host*\* and container*\* must share one endpoint for the fallback
  to work). M6.3's single-endpoint contract, sampler, and route are unchanged.
- **D7 — CPU per-mode collapse** to User/System at container granularity
  (cgroup has no iowait/irq/softirq); the 5-segment view is provided naturally by
  the host-level fallback (D2), not a separate toggle.
- **D8 — `ram_commit_*`** mapped best-effort at **host level** only (a
  node-level concept cloud reads from node_exporter); container level is honestly
  empty. Map only if vector's `host_metrics` memory collector actually emits a
  committed-memory series (plan-time pin); else the Memory-commitment chart stays
  honestly absent, as in M6.3.

## §4 Architecture

### §4.1 Collection (stack side, D1 deploy boundary unchanged)

- Add a **cAdvisor** service to `docker/docker-compose.logs.yml` (same operator
  opt-in overlay as M6.3's metrics additions). Run it **lean** to control
  cardinality:
  - `--store_container_labels=false`
  - `--whitelisted_container_labels=com.docker.compose.service,com.docker.compose.project`
  - `--disable_metrics=percpu,sched,process,hugetlb,referenced_memory,cpu_topology,resctrl,tcp,advtcp,udp`
    (keeps cpu / memory / network families; drops the noise).
  - exact flag set is a plan-time T1 verification against a re-captured lean fixture.
- Add cAdvisor's endpoint to vector's existing `prometheus_scrape` source, and a
  vector `filter`/`remap` transform that keeps **only Postgres-container series**
  and strips residual label noise, before the `prometheus_exporter` sink. Net:
  `:9598` now carries `host_*` (M6.3) + realtime/supavisor `service_*` (M6.3) +
  `container_*` (cAdvisor). No new registered endpoint.

### §4.2 Adapter dialect layer (`metrics.ts` — the D1 flip-cost extension point)

- A per-`stack_kind` source-name map (organized like `logflare-dialect.ts`):
  - `external` / `shared-db` (compose) → `container_*` (cAdvisor).
  - `k8s` → `container_*` (kubelet/cAdvisor) — **structure only, not bound**.
- Container filter lives in the **adapter**: select the series whose `name=`
  (or compose-service label) matches the project's registered `stack_meta.container`.
- **Additive fallback**: identity registered → emit container-level attrs from
  `container_*`; not registered → emit host-level attrs from `host_*` (M6.3
  behavior, byte-unchanged). CPU counter-delta rate math reused verbatim;
  per-mode collapses to user/system at container level.

### §4.3 Registry / identity (no new column)

- `stack_meta` (jsonb) gains `container` (compose: container name; k8s reserved
  `{namespace, pod_selector}` shape). No schema migration.
- parser (`projects-admin.ts`), register CLI
  (`docker/scripts/platform/register-project.ts`), and the GET `self_platform`
  block learn the key — mirroring the M6.3 `metrics_url` registration chain.
  Identity is non-secret (plain jsonb value), so no encryption/masking path.
- **shared-db**: children share the host's Postgres container → same `container`
  name → CPU/RAM is the shared container's (finer than VM, coarser than logical
  db). Panel hint, mirroring M6.3's shared-db note.

## §5 Light-up / honest degradation (per chart)

| Chart                                                    | Container-level (identity registered)                                                  | Fallback (not registered) |
| -------------------------------------------------------- | -------------------------------------------------------------------------------------- | ------------------------- |
| Memory usage                                             | `container_memory_working_set_bytes` absolute; `%` only if `spec_memory_limit>0`       | host\_\* RAM% (M6.3)      |
| CPU usage                                                | User / System (+`_other` remainder; no iowait/irq)                                     | host 5-segment (M6.3)     |
| Memory commitment                                        | `ram_commit_*` — host-level only if vector emits it, else honest-absent                | same                      |
| Disk usage                                               | **L1 SQL per-db (`pg_database_size` + WAL) — unchanged**                               | same                      |
| Disk IO / throughput                                     | **not container-charted** (cAdvisor fs/blkio = rootfs, misleading) → host\_\* or empty | host\_\*                  |
| Network                                                  | **`container_network_*` per-container (bonus)**                                        | host\_\*                  |
| Max-IOPS / throughput / Provisioned-disk reference lines | cloud compute-quota concept — honestly absent self-hosted                              | same                      |

## §6 Reuse / change boundary vs M6.3

- **Unchanged**: D1 deploy boundary (operator overlay + registered `metrics_url`
  - Studio consumes), the resident sampler loop + isolation, counter-delta rate
    math, `platform.metrics_samples`, the `infra-monitoring` route + response
    contract, the **entire L1 SQL half** (connections / db-size / WAL — already
    per-db), and the M6.3 `host_*` source (retained for the fallback).
- **Changed**: +cAdvisor service + vector `prometheus_scrape`/`filter`; the
  adapter's source-name constants → a `stack_kind`-keyed dialect + container
  filter + additive fallback; `stack_meta` +`container` key
  (parser/CLI/GET); +`ram_commit_*` host-level mapping (D8).
- This is the exact "L2 source swap only changes the adapter's source names +
  runbook; L1 and storage stay" that M6.3 D1's flip-cost column pre-committed.

## §7 Known trade-offs / accepted losses

- CPU per-mode collapses to User/System at container level (cgroup limit; D7).
- RAM% needs an operator-set container memory limit; absolute bytes otherwise (D4).
- No per-project disk-IO chart at container level (rootfs ≠ data volume; D4/§2.3).
- +1 container (cAdvisor) + its scrape cardinality (mitigated §4.1) (D1).
- k8s not implemented this milestone (D3).

## §8 Not in scope

- Per-container metrics for other services (kong/auth/storage/realtime) — M6.3 D4
  already scrapes realtime/supavisor `/metrics` for service-level signals; this
  milestone is about the **project's Postgres** resource footprint.
- k8s live implementation (D3, extension point only).
- Volume capacity/free-space collection (D4).
- daily-stats / billing semantics (unchanged from M6.3 — honestly absent).
