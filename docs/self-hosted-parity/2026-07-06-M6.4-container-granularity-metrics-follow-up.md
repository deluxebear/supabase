# M6.4 (candidate) — Container/Pod-granular infra metrics (follow-up to M6.3)

- **Status**: SUPERSEDED by the ratified spec `2026-07-06-M6.4-container-granularity-metrics-design.md` (2026-07-06). Kept as the discussion/spike record. Note: this exploratory draft assumed vector's `cgroups` collector; the T1 spike chose **cAdvisor** instead — see the design spec §2–§3 for the corrections.
- **Date**: 2026-07-06
- **Base**: custom/main @f811901fe7 (M6.3 merged)
- **Precedent**: M6.3 D1 (operator-deployed exporter, Studio consumes a registered `metrics_url`), M6.3 adapter + sampler (`lib/api/self-platform/metrics.ts`), M6.2/M6.3 dialect-layer pattern.

## §1 Motivation

M6.3 shipped **host-level, stack-scoped** infra metrics: vector `host_metrics`
reads the whole VM's `/proc`+`/sys`, so CPU/RAM/filesystem are one series per
stack. When multiple projects share a stack (shared-db children, or several
registered projects on one compose host), they all see the _same_ VM-global
numbers. Supabase Cloud avoids this by giving every project its own dedicated
EC2 instance, so "whole machine" == "this project" for free — a deployment
premise the management-plane does not have.

The better fix for self-hosted is to **drop the granularity from "machine" to
"the project's Postgres container/Pod"**. A project == one Postgres
container/Pod, so container-granular resource metrics are per-project by
construction, without needing dedicated machines. This also unifies compose and
k8s: both isolate containers with **cgroups**, so one collection approach covers
both. Target: make the Database observability tab (the same `database-charts.ts`
config M6.3 already feeds) report _this project's_ CPU/RAM rather than the VM's.

## §2 Spike findings (live, 2026-07-06, OrbStack stack, cgroup v2)

Measured against the running `supabase-db` container:

- **Container CPU/RAM are readable per-container.** `docker stats supabase-db` →
  CPU 3.10%, MEM 160.6 MiB. cgroup v2 files inside the container:
  `memory.current` = 179712000 (~171 MiB, gauge), `cpu.stat` `usage_usec`
  (cumulative µs counter). These are exactly what cAdvisor / vector's `cgroups`
  collector expose as `container_memory_working_set_bytes` /
  `container_cpu_usage_seconds_total`.
- **CPU per-mode degrades.** cgroup v2 `cpu.stat` carries only
  `user_usec`/`system_usec`/`nice_usec` — **no iowait/irq/softirq/idle**. Those
  are host-kernel concepts the container cgroup cannot see. So the 5-segment CPU
  chart (System/User/IOwait/IRQs/Other) collapses to at most User/System + total
  usage at container granularity. Accepted trade-off for per-project precision.
- **No memory limit set.** `supabase-db` runs with `Memory limit=0` /
  `memory.max=max`, so there is no container quota to use as the RAM-percent
  denominator (`docker stats` MEM% is vs. the whole VM). Unless the operator
  sets a container/Pod memory limit, RAM% at container granularity has the same
  "no real ceiling" problem host-level has — report absolute bytes, and only
  compute a percentage when a limit exists (`memory.max != max`).
- **vector can likely read cgroups without a new container.** vector
  `host_metrics` has a `cgroups` collector alongside the `cpu/memory/...` ones
  M6.3 already uses. If so, the collection-side change is _adding one collector
  to the existing M6.3 vector.yml_, not deploying cAdvisor — the single biggest
  simplification this spike found. (Exact metric names + 0.53 availability =
  this milestone's T1 spike, per the M6.3 fixture-is-binding protocol.)

## §3 Proposed shape (to be brainstormed/ratified if scheduled)

1. **Collection (stack side, D1 boundary unchanged).**
   - **Preferred**: extend M6.3's `docker/volumes/logs/vector.yml` `host_metrics`
     with the `cgroups` collector → same `:9598` exporter now also carries
     per-cgroup CPU/memory. Zero new containers; reuses the whole M6.3 pipeline.
   - **k8s**: cAdvisor is built into the kubelet and already emits `container_*`
     for every Pod — point `metrics_url` at it (or the cluster Prometheus). Same
     `container_*` naming family as vector's cgroups collector, so the adapter
     dialect is shared.
2. **Adapter dialect layer** (`metrics.ts`, the D1 flip-cost extension point).
   A per-`stack_kind` source-name map so `computeScrapeAttributes` reads
   `container_*` (cgroups/cAdvisor) instead of `host_*` (vector host_metrics),
   and **filters to the project's Postgres container** by name/label. CPU rate
   math (counter-delta) is reused verbatim; per-mode collapses to user/system.
3. **Registry**: `stack_meta` gains the project→container identity — a docker
   container name/label for compose rows, a `namespace`+pod-label-selector for
   k8s rows. No new column (stack_meta is jsonb); parser/CLI learn the keys.
4. **Light-up / honest degradation vs the M6.3 host-level surface**:
   - Memory usage, CPU usage (User/System only), disk IO/throughput, disk size:
     container-granular, per-project.
   - **Add the missing `ram_commit_*` mapping** (Memory commitment chart —
     cloud's node_exporter has `node_memory_Committed_AS_bytes`/`CommitLimit`;
     M6.3 left it unmapped). Cheap even at host level; do it here regardless.
   - CPU IOwait/IRQs segments: honestly empty at container granularity
     (cgroup can't provide them) — chart shows User/System/Other only.
   - RAM%: only when a container memory limit exists; else absolute bytes.
5. **Disk stays a hybrid** (unchanged reasoning from M6.3):
   - **Usage is already per-db and correct**: `pg_database_size` + WAL via L1
     SQL — independent of host/container, needs no change.
   - **Volume capacity/free** is the only gap: cAdvisor `container_fs_*` reads
     container rootfs, _not_ the Postgres data volume. k8s has clean per-PVC
     metrics (`kubelet_volume_stats_capacity_bytes`/`used_bytes`); compose needs
     the volume's host partition (or accept "no free-space ceiling"). The
     `disk_fs_size` reference line becomes best-effort/absent, like the cloud
     "Provisioned disk size" line is absent self-hosted.

## §4 Reuse / change boundary vs M6.3

- **Unchanged**: D1 deploy boundary (operator exporter + registered
  `metrics_url` + Studio consumes), the sampler loop + isolation, counter-delta
  rate math, `platform.metrics_samples`, `infra-monitoring` route + response
  contract, and the **entire L1 SQL half** (connections, db-size, WAL — already
  per-db).
- **Changed**: the `host_metrics` collector set (add `cgroups`) or exporter
  choice (cAdvisor on k8s); the adapter's source-name constants → a
  stack*kind-keyed dialect + container filter; `stack_meta` gains the container
  identity; `ram_commit*\*` mapping added.
- This is the exact "L2 source swap only changes the adapter's source names +
  runbook; L1 and storage stay" that M6.3 D1's flip-cost column pre-committed.

## §5 Known trade-offs / open questions (decide at brainstorm)

- **CPU per-mode loss** (cgroup has only user/system/nice). Accept the collapse,
  or keep an optional host-level 5-segment view for single-project stacks?
- **RAM% denominator**: require operators to set container memory limits (so
  `memory.max` gives a ceiling), or always show absolute bytes + only
  percentage-when-limited?
- **shared-db**: multiple projects on one Postgres container → CPU/RAM is that
  shared container's, still not per-logical-database (finer than VM, coarser
  than db). Panel hint like the M6.3 shared-db note.
- **Volume free-space** on compose: worth solving, or accept usage-only (L1)
  - no capacity ceiling?
- **Collector confirmation**: does vector 0.53's `cgroups` collector exist and
  what are its exact series names? (Binding T1 spike; capture a fixture like
  M6.3's `metrics-scrape.prom`.) If not, fall back to a cAdvisor overlay.

## §6 Not in scope

Per-container metrics for the _other_ services (kong/auth/storage/realtime) —
M6.3's D4 already scrapes realtime/supavisor's own `/metrics` for service-level
signals; this follow-up is specifically about the **project's Postgres**
resource footprint, which is what the Database observability tab is about.
