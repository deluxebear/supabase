## §1 Problem and goal

Today creating a project on the self-hosted platform is manual and two-step: (a) hand-provision
infrastructure (a full second compose stack, or `docker exec … psql -c "create database …"` on the
shared stack), then (b) run `docker/scripts/platform/register-project.ts` over SSH. There is no
UI entry: `POST /api/platform/projects` does not exist (`pages/api/platform/projects/index.ts` is
GET-only) and `DELETE /api/platform/projects/[ref]` does not exist either.

M5.0 delivers, for the compose track:

1. **Quick-create** from Studio: new database on an existing shared stack, registered and usable
   immediately (Postgres-level data isolation, M2 semantics).
2. **Attach existing stack** from Studio: register a fully independent stack by filling in
   connection details, with a live connectivity check.
3. **Delete** from Studio: deregister (row removal only), Owner-only, confirm-by-typing-ref.
4. Registry metadata (`stack_kind`, `stack_meta`) so the M5.1 k8s track slots in without schema
   or contract changes.

## §2 Recon facts (verified in source)

- `platform.projects` — 22 columns, all connection params + 6 encrypted credential columns
  (`docker/volumes/platform/migrations/02-projects.sql`). No stack metadata today.
- Register CLI — `docker/scripts/platform/register-project.ts` (275 lines): `register` (explicit
  flags or `--from-current-env`), `deregister`, `list`; writes via
  `docker exec supabase-platform-db psql`; guards empty-secret registration
  (`REQUIRED_INPUT_FIELDS`).
- Upstream create mutation posts cloud-shaped `CreateProjectBody` to `POST /platform/projects`
  (`apps/studio/data/projects/project-create-mutation.ts:80`). Our self-platform form does NOT
  reuse it (cloud body is billing/region-coupled); we add a self-platform mutation with our own
  body (M1 precedent: self-platform shapes already diverge from the cloud OpenAPI contract).
- Upstream delete mutation calls `del('/platform/projects/{ref}')`
  (`apps/studio/data/projects/project-delete-mutation.ts:16`) — exactly the route we add, so the
  delete mutation + its cache invalidation are reused as-is.
- `pages/new/index.tsx` is the new-ORG wizard (untouched); `pages/new/[slug].tsx` is the
  new-project wizard — interception point for the self-platform form.
- FKs referencing `platform.projects`: `role_projects.project_id … on delete cascade`
  (`04-roles.sql:19`); `auth_config.project_ref … on delete cascade` (`06-auth-config.sql:11`).
  Invitations store `role_scoped_projects` as **text refs, re-validated at accept**
  (`05-invitations.sql:12-14`) — a project deleted between invite and accept fails loud at
  accept; no FK blocks row deletion.
- RBAC: `guardOrgRoute`/`guardProjectRoute` + `checkPermissionWithContext` MFA chokepoint
  (`lib/api/self-hosted/enforce.ts`); matrix `%` wildcard gives Owner/Admin all actions; the
  Owner-protection **restrictive deny** is the precedent for carving Admin out of a specific
  action (`lib/api/self-hosted/matrix.ts`).
- Secrets: `lib/api/self-hosted/secrets.ts` `encryptSecret`/`decryptSecret`,
  `PLATFORM_ENCRYPTION_KEY`, fail-closed. Studio server already holds the key (resolver decrypts
  at request time since M2).
- Per-ref SQL channel: `executeQuery(sql, { projectRef })` threads the resolved connection to the
  pg-meta proxy (M2 Task 8). PostgreSQL allows `CREATE DATABASE` as the sole statement of a
  simple-query message (autocommit, no transaction block) — **plan Task 2 opens with a live spike
  confirming the pg-meta channel accepts it** (M2 E2E created databases via `docker exec psql`,
  so this exact path is unexercised).
- Analytics columns (`logflare_url_enc`/`logflare_token_enc`, 03-analytics) are per-ref with
  strict no-fallback (M2.1): quick-create must NOT clone them (see §4).

## §3 Data model — migration `07-stack-metadata.sql`

```sql
-- fresh initdb AND manual apply on existing platform-db (see README upgrade paragraph)
alter table platform.projects add column if not exists stack_kind text not null default 'external';
alter table platform.projects add column if not exists stack_meta jsonb not null default '{}';
```

- `stack_kind` ∈ `'external'` (independent stack, attached/registered) | `'shared-db'` (database
  on another registered project's server) | `'k8s'` (M5.1). App-level validation; no CHECK
  constraint (consistent with the unconstrained `status` column; the write paths are the POST
  route and the CLI, both validate).
- `stack_meta`: `shared-db` rows store `{"host_ref": "<ref>"}`; `k8s` rows (M5.1) will store
  `{"namespace": …, "release": …}`; `external` rows `{}`.
- Backfill: existing rows default to `'external'`. Pre-existing shared-db-style rows (e.g.
  `proj-b`) may be relabeled manually; the difference is informational in M5.0 (host-stack
  dropdown filter + display) — README documents the optional `update` statement.
- Idempotent, follows 03-analytics `add column if not exists` precedent; header carries the
  fresh-initdb-AND-manual-apply cross-ref like 04/05/06.
- Register CLI: new optional `--stack-kind` flag (validated against the three kinds, default
  `'external'`); `stack_meta` not settable via CLI in M5.0 (no use case).

## §4 Create API — `POST /api/platform/projects`

`pages/api/platform/projects/index.ts` gains a POST branch. GET (V1 legacy + V2) is untouched.
In plain self-hosted mode POST returns 404 `{message: 'Not available on this deployment'}`
(house pattern); GET keeps its legacy `[DEFAULT_PROJECT]` behavior — the zero-break sibling
asserts both.

Body (self-platform-only contract):

```ts
// mode 'shared-db' — quick-create
{ mode: 'shared-db', organization_slug: string, name: string, ref: string, host_ref?: string /* default 'default' */ }

// mode 'external' — attach existing stack
{ mode: 'external', organization_slug: string, name: string, ref: string,
  connection: {
    dbHost: string, dbPass: string, kongUrl: string,
    anonKey: string, serviceKey: string, jwtSecret: string,          // required
    dbPort?: number /* 5432 */, dbName?: string /* 'postgres' */,
    dbUser?: string /* 'supabase_admin' */, dbUserReadonly?: string /* 'supabase_read_only_user' */,
    restUrl?: string /* derived: kongUrl minus trailing slash + '/rest/v1/' (CLI precedent) */,
    publishableKey?: string, secretKey?: string,
    logflareUrl?: string, logflareToken?: string,
  } }
```

Validation (400, before guard — recorded order deviation, same class as M3.1 PATCH: the guard
needs `organization_slug` from the body):

- `mode` ∈ {`shared-db`, `external`}; `organization_slug` present; `name` non-empty ≤ 64 chars.
- `ref` matches `^[a-z][a-z0-9-]{2,29}$` and is not the reserved `'default'`. The UI
  auto-suggests a slug from `name`; the API always requires an explicit ref.
- `shared-db`: `host_ref` must resolve to an existing row with `stack_kind = 'external'`
  (else 400 with a message naming the constraint).
- `external`: required connection fields present (mirror of the CLI's
  `REQUIRED_INPUT_FIELDS` guard).

Guard: `guardOrgRoute` with `action: write:Create (PermissionAction.CREATE)`,
`resource: 'projects'`, org from `organization_slug`. Matrix `%` wildcard ⇒ Owner/Admin allowed,
Developer/Read-only 403. No matrix change. MFA enforced via the existing
`checkPermissionWithContext` chokepoint.

### shared-db flow (insert-first, no compensating DROP)

1. Load host row (`host_ref`); derive `db_name = ref` with `-` → `_`.
2. **Insert the new row first** with `status = 'COMING_UP'`: clone from the host row
   `db_host, db_port, db_user, db_user_readonly, kong_url, rest_url, cloud_provider, region` and
   all 6 credential ciphertexts **verbatim** (same encryption key; shared-stack semantics are the
   point). `logflare_url_enc`/`logflare_token_enc` are **NOT cloned — left NULL** so analytics
   honestly 404s "not configured" (M2.1 no-fallback) instead of showing the host stack's logs.
   Set `organization_id` from the guard's org context, `stack_kind = 'shared-db'`,
   `stack_meta = {"host_ref": …}`. The `ref` UNIQUE constraint is the concurrency gate — a
   duplicate ref fails here, race-free, before any DDL (409 mapped from the unique violation).
3. `CREATE DATABASE "<db_name>"` via `executeQuery` against `host_ref`'s connection —
   single-statement, strict-regex-validated identifier in double quotes (DDL identifiers cannot
   be parameterized; the regex from validation step is the injection barrier).
4. Success → flip row to `ACTIVE_HEALTHY`, return 201. DDL failure → delete the row, return 500
   with the pg error message. Crash between 2 and 4 leaves a visible `COMING_UP` row the user can
   remove with the delete endpoint (honest, self-serviceable).

Read-only DSN: `supabase_read_only_user` + `pg_read_all_data` are cluster-level, so the new
database is covered with zero extra grants (proven live in M3.0 E2E PASS 4).

### external flow

1. Assemble the candidate connection from the body (defaults as above).
2. **Connectivity probe**: run `select 1` through the existing pg-meta proxy channel with the
   candidate DSN passed explicitly (new `probeConnection` helper in the data layer reusing the
   same header mechanism `executeQuery` uses; NO new database driver dependency). Failure → 400
   `{message: 'Could not connect to database: <cause>'}`. This is a real improvement over the
   CLI, which registers blind (README's "registration quality" caveat class).
3. Encrypt `dbPass, serviceKey, anonKey, jwtSecret` (+ optional publishable/secret/logflare
   values) with `encryptSecret` — fail-closed if `PLATFORM_ENCRYPTION_KEY` is missing (500, no
   row).
4. Insert with `status = 'ACTIVE_HEALTHY'`, `stack_kind = 'external'`, `stack_meta = {}` → 201.

Response (201, both modes): `{ id, ref, name, status, organization_slug }` — enough for the UI to
invalidate the project-list queries and `router.push('/project/<ref>')`.

## §5 Delete API — `DELETE /api/platform/projects/[ref]`

`pages/api/platform/projects/[ref]/index.ts` gains a DELETE branch (GET untouched; Allow header
extended). Plain mode: DELETE → 404 house message, GET byte-identical to today.

Order: ghost ref → 404 (`guardProjectRoute` resolves first, 404-before-403 by construction) →
guard 403 → `ref === 'default'` → 400 `{message: 'The default project cannot be deleted'}` →
business.

Guard: `guardProjectRoute` with `action: write:Delete (PermissionAction.DELETE)`,
`resource: 'projects'`. **Owner-only**: one restrictive deny row added to the matrix denying
Admin `write:Delete` on `projects` (Owner-protection precedent; test-pinned both directions:
Owner 200, Admin 403). Flip cost to cloud-parity (Admin allowed) = delete that row.

Business (two sequential parameterized statements via `executePlatformQuery` — same-snapshot
CTE semantics forbid folding them, M1 I1-BUG class):

1. `delete from platform.projects where ref = $1` — cascades take `role_projects` scoping rows
   (04-roles) and `auth_config` (06-auth-config) atomically.
2. GC derived roles orphaned by that cascade: delete project-scoped (derived) roles that now have
   zero `role_projects` rows, using the same derived-role predicate as the M3.1 GC (the
   `isOrgScopedRole` complement; exact SQL pinned in the plan against 04-roles.sql). Crash window
   between 1 and 2 leaves a **zero-grant** orphan derived role — fail-closed by the M3.1 I1
   guards (empty-scope derived roles grant nothing), noted in README.

Response: 200 `{ ref }`. The upstream delete mutation handles the rest (cache invalidation,
toast). Pending invitations scoped to the deleted project fail loud at accept (ref re-validation,
05-invitations by design). The real database is untouched — response body and README both say so.

## §7 Zero-break boundaries

- Plain self-hosted (both flags false): POST + DELETE → 404 `Not available on this deployment`
  byte-identical to unit siblings; GET behaviors unchanged; `/new/[slug]` renders the upstream
  wizard exactly as today (early return not taken).
- Cloud/hosted mode: untouched (all new code inside `IS_SELF_PLATFORM`).
- Existing registered projects: backfilled `stack_kind='external'` — resolver reads none of the
  new columns; behavior identical.
- Register CLI remains fully supported (ops/bootstrap path); README positions UI as the primary
  path and CLI as fallback.

## §8 Error handling summary

| Case                                                   | Response                                                                            |
| ------------------------------------------------------ | ----------------------------------------------------------------------------------- |
| invalid mode / missing fields / bad ref / reserved ref | 400 with specific message                                                           |
| duplicate ref (either mode)                            | 409 mapped from unique violation                                                    |
| host_ref missing or not `external`                     | 400 naming the constraint                                                           |
| CREATE DATABASE fails                                  | row deleted, 500 with pg error                                                      |
| probe fails (attach)                                   | 400 `Could not connect to database: <cause>`                                        |
| PLATFORM_ENCRYPTION_KEY missing                        | 500 fail-closed, no row                                                             |
| delete ghost ref                                       | 404 `Project not found`                                                             |
| delete `default`                                       | 400                                                                                 |
| delete as Admin/Developer                              | 403 (matrix deny / no grant)                                                        |
| crash windows                                          | COMING_UP orphan row (user-deletable); zero-grant orphan derived role (fail-closed) |

## §10 M5.1 outlook (designed-for, not built)

- k8s rows are still self-contained (`db_host` → `<release>-rw.<ns>.svc` or NodePort DSN;
  `stack_kind='k8s'`, `stack_meta={"namespace","release"}`).
- Provisioner = a control-plane service (Helm SDK or `helm` exec) driving: namespace → Secrets
  (generated JWT/anon/service/db-pass) → CNPG cluster → chart install → readiness → write-back
  row + flip `COMING_UP` → `ACTIVE_HEALTHY`. POST route goes async for `mode: 'k8s'` — status
  polling already contract-native.
- `apply-auth-config` gains a k8s apply target (patch GoTrue Deployment env) — M4's store/apply
  separation pays off here.
- CNPG then carries F4 (backups/PITR via S3 WAL archive) and F5 (read replicas).

## §11 Risks and accepted limitations

1. **pg-meta CREATE DATABASE unexercised** — plan Task 2 opens with a live spike; if the channel
   refuses non-transactional DDL, fallback is a dedicated single-statement code path in the same
   proxy (design assumption LOW risk: PG allows it in autocommit; pg-meta executes raw SQL).
2. **shared-db inherits the M2 boundary** — shared gateway/API keys/JWT secret; GoTrue/Storage
   bound to the host's main database. README cross-references the existing M2 boundary section;
   the create UI shows a one-line callout.
3. **stack_kind backfill mislabels pre-existing shared-db-style rows** as `external` —
   informational only; README documents the optional relabel UPDATE.
4. **Orphan artifacts on crash windows** (§8) — both fail-closed and self-serviceable.
5. **No DROP DATABASE anywhere in M5.0** — deleting and recreating a project with the same ref
   quick-create will fail at CREATE DATABASE (already exists → 500 with pg message). Acceptable:
   the message is explicit; README documents manual DROP as the remedy.
6. **Attach probe uses the rw DSN only** — read-only-user misconfiguration (M3.0 E2E ENV FIX
   class) still surfaces late. Probing both DSNs is a cheap M5.x follow-up; not blocking.
