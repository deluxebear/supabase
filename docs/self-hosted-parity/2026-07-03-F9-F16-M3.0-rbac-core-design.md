## 3. 硬约束（M2.2 终审移交，必须遵守）

### 3.1 Shared-stack JWT secret（本 spec 显式引用，约束 #1）

引用 `docker/volumes/platform/README.md` "Shared-stack JWT secret" 段（README:564-581）：**同一底层栈上注册的多个项目共用该栈的 JWT 签名密钥；为某 ref 验签有效的 JWT——包括 `temporary` 端点为它铸的短期 JWT、`config/index`/`config/postgrest` 为它返回的 `jwt_secret`——对同栈兄弟项目在密码学上同样有效。**

对 M3.0 设计的直接推论：

1. **RBAC 的执行面 = 平台 API 层，主体 = platform GoTrue 会话（`claims.sub`）**。绝不以"调用方持有某 ref 的数据面凭据"作为授权或隔离依据——该凭据在同栈兄弟项目上同样有效。
2. **凭据可见 ⇒ 同栈全部兄弟项目的数据面钥匙**。因此凭据类路由（§7.3 Class C）收紧到 org 管理级：把"能看到 jwt_secret/service key"显式建模为跨项目信任，而不是假装它是项目内权限。project_scoped 角色（M3.1 起可指派）不授予凭据类路由访问。
3. **会话级只读（§7.4）与本边界同性质**：抬高门槛、防误操作，不是密码学/内核级硬隔离。两者在 README 同一节内如实声明；真隔离的路径（独立栈、per-project 只读 DB 角色）作为后续工作记录。

### 3.2 沿用铁律

`packages/api-types` 为唯一契约真相源（只读）；禁 `as any`（DB text→枚举窄化除外）；所有新行为以 `IS_SELF_PLATFORM` 门禁；纯自托管零破坏为 HARD 约束（sibling `*.self-hosted.test.ts`，断言须全字节/可注错，不得只查 status）；resolved 行唯一凭据源不回退（M3.0 不触碰该不变量，新增 enforcement 只叠加在其之上）；错误体顶层 `{ message }`（403 同样）；零新增 npm 依赖（评估器仅依赖已有 `json-logic-js`）。

## 4. 数据模型：`04-roles.sql`

新迁移 `docker/volumes/platform/migrations/04-roles.sql`，**幂等**（`create table if not exists` + 种子/回填 `on conflict do nothing`，03 先例），README 增补手工升级段（`docker exec -i supabase-platform-db psql ... < 04-roles.sql`，README:218-229 先例）。

```sql
platform.roles (
  id               bigint generated by default as identity primary key,
  organization_id  bigint not null references platform.organizations(id) on delete cascade,
  base_role_id     bigint not null references platform.roles(id),   -- 基础角色自指（= id）
  name             text   not null,
  description      text,
  created_at/updated_at timestamptz default now(),
  unique (organization_id, name)
)
platform.role_projects (        -- project_scoped 派生角色的项目集（M3.1 起写入；M3.0 展开器即读它）
  role_id     bigint not null references platform.roles(id) on delete cascade,
  project_id  bigint not null references platform.projects(id) on delete cascade,
  primary key (role_id, project_id)
)
platform.member_roles (
  profile_id  bigint not null references platform.profiles(id) on delete cascade,
  role_id     bigint not null references platform.roles(id) on delete cascade,
  created_at  timestamptz default now(),
  primary key (profile_id, role_id)
)
```

- **种子**：default org 的 4 个基础角色，显式 id 1-4（`generated by default` + 显式 insert + `setval`），`base_role_id = id` 自指，顺序与名称**逐字对齐前端 `FIXED_ROLE_ORDER` 常量**（预期 `Owner / Administrator / Developer / Read-only`；任务实现时以源码为准核对，写进测试断言）。
- **回填**：`insert into member_roles (profile_id, role_id) select om.profile_id, 1 from organization_members om ... on conflict do nothing` —— 存量成员全部 Owner（决策 #8）。
- **首登路径不变**：`createProfileWithDefaultMembership` 仍只建 profile + organization_members（无角色）——新用户零角色 fail-closed（决策 #3）。M3.2 邀请制落地后由邀请携带 role_id。
- 派生角色（`base_role_id ≠ id`、`role_projects` 非空）的**创建**属于 M3.1（PATCH member V2 隐式创建，前端无独立端点——recon §4）；M3.0 建表 + 展开器/可见性读它（psql 手工插入即可 E2E），不算提前建空表。
- `organization_members` 表保持不动（成员资格与角色持有分离，云端 Member.role_ids 语义即多对多）。

## 5. 角色→权限矩阵 v1（代码常量，权威表）

落点 `apps/studio/lib/api/self-platform/rbac/matrix.ts`。每个基础角色 → 授权模板数组 `{actions, resources, condition?, restrictive?}`（形状即 `AccessControlPermission` 去掉 org/project 字段，展开时补齐）。动作枚举全集 39 个见 recon notes §2（`@supabase/shared-types` `PermissionAction`）。

| 角色              | 授权模板（v1）                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| ----------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Owner**         | `{actions:['%'], resources:['%'], condition:null}`                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| **Administrator** | `{actions:['%'], resources:['%'], condition:null}` 加两条限制：① `{actions:['write:%'], resources:['organizations'], restrictive:true}`（组织级变更留 Owner）；② `user_invites` / `auth.subject_roles` 的 CREATE/DELETE 授权携带 json-logic condition `{'!==':[{var:'resource.role_id'}, 1]}`（不得授予/撤销 Owner；M3.1 端点与前端 `rolesAddable/rolesRemovable` 同一 condition 生效，recon §4）                                                                                                                                                        |
| **Developer**     | 读全集（见下）+ 内容写：`tenant:Sql:%`、`sql:%`、`storage:Write`、`storage:Admin:Write`、`functions:Write`、`functions:Secret:Read/Write`、`auth:Execute`、`analytics:Write`，另加一条内容资源授权 `{actions:['write:%'], resources:['user_content'], condition:null}`（SQL 片段等用户内容的增删改，云端 Developer 语义）。**不含**：`write:Create/Update/Delete`（控制面对象——`user_content` 例外见上）、`secrets:Read/Write`（凭据类，§7.3）、`infra:Execute`、`billing:%`、`analytics:Admin:Write`、`realtime:Admin:Write`、`replication:Admin:Write` |
| **Read-only**     | 读全集：`read:Read`、`analytics:Read`、`analytics:Admin:Read`、`functions:Read`、`sql:Read:Select`、`tenant:Sql:Query`、`tenant:Sql:Read:Select`、`tenant:Sql:Admin:Read`、`storage:Read`、`storage:Admin:Read`、`realtime:Admin:Read`、`replication:Admin:Read`（`tenant:Sql:Query` 保留给 SQL Editor 跑 SELECT，写由 §7.4 会话级只读拦截）                                                                                                                                                                                                             |

- 矩阵为 **v1 权威表**：实现期发现与前端 gate 实际用的 (action, resource) 组合不符时，修订须经 reviewer 核对并记入 progress.md（不得静默改）。
- 通配符语义与评估器一致（`%`→`.*` 全锚定，授权侧持模式）。resources 维度 v1 以 `['%']` 为主 + 上表点名的例外，避免发明云端未知的资源名清单；后续按需细化。

## 6. 权限展开器与 `GET /platform/profile/permissions`

新模块 `lib/api/self-platform/rbac/expand.ts`：

- `getMemberContext(gotrueId)`（落 `lib/api/self-platform/members.ts`）：单次 join 查询 profiles → organization_members → member_roles → roles → role_projects（+projects 拿 ref/id/name），返回 `{ profileId, orgs: [{ orgId, orgSlug, roles: [{id, baseRoleId, name, projectRefs, projectIds}] }] }`。M3.1 的 members/roles 端点复用同一数据层。
- `expandPermissions(memberContext)`：每持有角色 × 矩阵模板 → `AccessControlPermission[]`。org_scoped（`base_role_id === id`）→ `project_refs: []`、`project_ids: []`；派生角色 → 模板取 **base 角色**的，`project_refs/project_ids` 填 `role_projects` 关联的项目。`organization_id/organization_slug` 按角色所属 org 填。零角色 → `[]`。
- `permissions.ts` 路由：通配实现整体替换为 `claims.sub → getMemberContext → expandPermissions`；响应类型对 `components['schemas']['AccessControlPermission'][]` 编译期断言（注意 nullable 字段族与必填 `organization_slug`，recon §3）。
- 前端感知：`permissions-query` staleTime 5 分钟（recon §1）——角色变更后 UI gate 最迟 5 分钟刷新；服务端 enforcement 实时，不受影响。记 README 一句话。

## 7. 服务端 enforcement

### 7.1 评估器抽纯（动 1 个上游文件）

`doPermissionsCheck` + `doPermissionConditionCheck`（`hooks/misc/useCheckPermissions.ts:14-66`）是纯函数，唯一依赖 `json-logic-js`（已有依赖）。抽到新文件 `apps/studio/lib/permissions-check.ts`（无 React 依赖），`useCheckPermissions.ts` 改为从新文件 import（行为零变化，`[self-platform]` 注释标记，登记上游同步保护清单）。服务端与前端从此共用同一份评估逻辑——"服务端 RBAC 复用 doPermissionsCheck"以字面成立。

### 7.2 `requirePermission` helper

`lib/api/self-platform/rbac/enforce.ts`：

```ts
requirePermission(claims, { action, resource, projectRef? }): Promise<
  { ok: true; ctx: MemberContext } | { ok: false; status: 403; message: 'Forbidden' }>
```

- 流程：`claims.sub` → `getMemberContext`（每请求一次 platform-db 查询，内部平台规模无缓存需求）→ `expandPermissions` → `doPermissionsCheck(perms, action, resource, data?, orgSlug, projectRef)`。
- 与 resolver 的次序：**404 先于 403**——未知 ref 仍由 `resolveProjectConnection`/`ProjectNotFound` 得 404 `{message:'Project not found'}`（现状语义不变）；已知 ref 无权限 → 403 `{message:'Forbidden'}`。
- 单组织现状下 orgSlug 由 memberContext 唯一 org 得出；签名按多组织契约留参。
- 仅 `IS_SELF_PLATFORM` 路径调用；plain self-hosted 分支字节不变。

### 7.3 路由分级表（M3.0 enforcement 覆盖面，决策 #2/#4）

| Class                                 | 路由族                                                                                                                                                                                                                                 | 校验                                                                                                                                                                                                                                                                                                                                                                           |
| ------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **C 凭据类**（Owner/Admin）           | `projects/[ref]/config/index`、`config/postgrest`、`api-keys/*`（含 by-id、temporary）、`settings` 的 `service_api_keys` 字段                                                                                                          | `requirePermission(action:'secrets:Read'（写类端点 secrets:Write）, resource:'projects', projectRef)` —— 矩阵只授 Owner/Admin ⇒ Developer/Read-only/项目级角色 403。`settings` 与 `props/project/[ref]/api` 采用**响应过滤**而非整路由 403：非 admin 层调用者拿到 service/secret 字段置空（anon/publishable 保留，页面不塌）——前端对置空值的容忍度列为任务内验证点（RED 先行） |

| **D 任意 SQL**（成员即可 + 只读会话） | `pg-meta/[ref]/query` 及 `executeQuery` 消费者（lints、migrations 读路径）                                                                                                                                                             | `requirePermission('tenant:Sql:Query', 'projects', ref)`（Read-only 有此授权）+ §7.4 只读会话                                                                                                                                                                                                                                                                                  |

- 全部 [ref] 路由的声明清单在 writing-plans 阶段逐文件枚举（M2.1 的 17 个 factory 消费者清单是现成底稿）；M3.0 内完成上表五类，若有漏网路由由 §10 的 grep 守卫 + 终审兜底，补进 M3.1。
- 组织级端点：M3.0 现存的 organizations 读端点走 §8 可见性；成员管理写端点属 M3.1（届时用同一 helper + `user_invites`/`auth.subject_roles` condition）。

### 7.4 Read-only 会话级只读（决策 #5）

- 机制：Class D 校验通过后，若调用者对该 ref 的**有效基础角色为 Read-only**（多角色取最高权），`executeQuery` 侧对该次连接串附加 `options=-c default_transaction_read_only=on` 再加密传给 pg-meta。SELECT 正常；INSERT/UPDATE/DDL 得 PG 原生 `cannot execute ... in a read-only transaction` 错误（经现有错误通道透传）。
- **诚实边界（README 与本 spec 双记）**：会话级默认可被刻意 `BEGIN READ WRITE` / `SET transaction_read_only=off` 绕过——这是防误操作与 UI 语义一致性，不是硬隔离；与 §3.1 shared-stack 边界同节声明。硬隔离 = per-project 只读 DB 角色进注册表（明确记为后续工作，不在 M3）。
- **技术验证点（首任务 spike 步骤）**：DSN `options` 参数经 pg-meta（node-postgres）的透传行为——若不透传，回退方案（如 role-aware 连接串走独立凭据列）升级为设计变更，回到人工决策，不得静默换方案。

## 8. 可见性过滤

- 两个项目列表路由（`projects/index.ts`、`organizations/[slug]/projects.ts`）handler 增加 `claims` 第三参（`apiWrapper` 已传，profile 路由先例），把调用者可见项目集下推到 `listAllProjectsV2/listOrgProjectsV2` → `queryProjectRows` 的 SQL suffix（`where organization_id in (成员 org) [and id in (指派项目)]`）。持任意 org_scoped 角色 → 见该 org 全部项目；仅持派生角色 → 只见 `role_projects` 指派项目；零角色 → 空列表（分页 count 同步为 0）。
- `projects/[ref]/index.ts` 详情路由同样过 Class V 校验（不可见 ref → 403；未注册 ref 仍 404——先 404 后 403 次序见 §7.2）。
- `organizations` 列表/详情：按成员资格过滤（非成员 org 不可见）；`is_owner` 由"持有 Owner 基础角色"真实化（替换 `organizations.ts:23` 硬编码 `true`）。
- **空注册表 default 回退**保持：回退仅对"持任意角色的成员"生效，零角色用户即使空注册表也得空列表（fail-closed 一致性）；plain self-hosted 不受影响（zero-break sibling 覆盖）。
