# M0.3 Customized Supabase Postgres Kubernetes Replacement Spike

## Result

PG17: PASS on 2026-07-12.

OrioleDB17: PASS on 2026-07-12 after the locally available tag was refreshed.
The current `deluxebear/postgres:orioledb-17` image exposes managed pgBackRest
2.58.0. Its fixture also requires the Supabase `supabase_admin` role and
`orioledb` extension initialization, both now covered by preflight.

## Decision

The default Kubernetes recovery path uses the customized Supabase PostgreSQL
image and its built-in pgBackRest implementation. It does not install or
require CloudNativePG, cert-manager, Barman Cloud, or CNPG-I.

CloudNativePG remains an optional post-core provider for deployments that
already use it. It is not a dependency of the default self-hosted Supabase
deployment.

## Scope

The destructive spike is implemented in
`apps/backup-operator/spikes/kubernetes-replacement.sh`. It creates one uniquely
named Kind cluster and deletes it on exit. The only optional argument is a
closed allowlist selecting `pg17` or `orioledb17`; arbitrary images, SQL,
resource names, credentials, and paths are not accepted.

The spike verifies:

1. a source StatefulSet runs `deluxebear/postgres:17` on its own PVC;
2. the image-managed pgBackRest 2.58.0 binary creates the stanza, checks WAL
   archiving, and takes a physical full backup;
3. data is written before and after a named restore point;
4. a Kubernetes Job restores the named target into a separately allocated PVC;
5. a replacement StatefulSet starts from the restored PVC;
6. an isolated Service sees only the data present at the recovery target;
7. the replacement database has the same PostgreSQL system identifier but a
   different PVC UID;
8. the stable Service switches to the replacement workload and then rolls back
   to the source workload;
9. the source PVC UID, PV binding, Pod UID, and post-target data remain
   unchanged through restore, cutover, and rollback.

## Provider constraints

- The Kubernetes provider acts only on explicitly self-managed StatefulSets.
- Recovery always targets a newly allocated PVC; it never edits the source PVC.
- The PostgreSQL system identifier is preserved because the replacement belongs
  to the same database history, while the pgBackRest stanza/archive identity
  must be changed before post-cutover archiving is enabled.
- Service endpoint propagation is asynchronous. Cutover and rollback are not
  complete merely because a query succeeds; the observed database history must
  match the expected target.
- The provider must positively detect the managed pgBackRest binary and reject
  incompatible image variants before scheduling a destructive job.
- The source workload and PVC remain quarantined and available for explicit
  rollback until a separately audited cleanup operation removes them.
