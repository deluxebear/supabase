## 1. Context and goal

M2 introduced the project registry (`platform.projects`) and the single-entry resolver
`resolveProjectConnection(ref)` (`apps/studio/lib/api/self-platform/resolve-connection.ts`),
but only wired it into a subset of `[ref]` routes: pg-meta query, project detail/databases/
settings, and v1 api-keys. Everything else still reads global env regardless of the `ref`
path parameter. Consequences today:

- **Credential leakage (client-facing):** `pages/api/platform/projects/[ref]/api-keys/temporary.ts`
  returns the permanent global `SUPABASE_SERVICE_KEY` for any ref to any authenticated user
  (live consumer: Realtime Inspector via `data/api-keys/temp-api-keys-query.ts`).
  `pages/api/platform/props/project/[ref]/api.ts` returns global anon + service keys for any
  ref (no live frontend consumer, but the exposure is real).
- **Wrong-stack proxying (server-side):** all 7 auth-admin routes, all 14 storage routes, and
  all 3 analytics routes operate on the default stack's GoTrue/Storage/Logflare for every ref.
- **Decorative pagination:** `list-user-projects.ts` echoes limit/offset without slicing;
  the data layer has no LIMIT/OFFSET.
- **Typing debt:** project-list responses use bespoke interfaces with no compiler tie to
  `packages/api-types`.

M2.1 makes every remaining `[ref]` surface resolve through the registry, closes both
credential-bearing routes, and pays down the pagination/typing debt. It is a hardening
milestone: no new UI, but it is the isolation foundation M3 (teams/roles RBAC) requires â€”
RBAC over keys and data planes is meaningless while any ref reaches the default stack.

## 3. Hard invariants (carried from M1/M2)

- **Resolved row is authoritative:** once `resolveProjectConnection(ref)` hits a registry row,
  every value comes from that row. Never fall back to global env for individual fields â€”
  including the new analytics fields (NULL â†’ honest "not configured", never default-stack data).
  Global env serves exactly one case: unregistered `default`.
- **Plain self-hosted zero-break is HARD:** with `IS_SELF_PLATFORM` false, behavior must be
  byte-identical. Every touched route gets/keeps a sibling `*.self-hosted.test.ts`
  (dynamic-import pattern from M2 Task 6).
- **Dual-key model:** `PLATFORM_ENCRYPTION_KEY` for at-rest registry secrets (crypto-js, via
  `secrets.ts`), `PG_META_CRYPTO_KEY` for pg-meta transport. New `logflare_token_enc` uses the
  at-rest key.
- **Contracts from `packages/api-types`;** no `as any` (enum narrowing `status as X['status']`
  remains the sole sanctioned `as` exception).
- Sanctioned deviations carried: crypto-js in root `package.json`; `databases.ts` plain
  self-hosted `localhostâ†’AWS` display mapping.

## 4. Design

### 4.1 Mechanism: helper-level resolution + centralized 404 (chosen approach)

Follow the M2 Task 8 precedent (`executeQuery(projectRef)`): helpers accept a ref and resolve
internally, so the 17 admin-client routes change by roughly one line each. Complemented by one
centralized error mapping: `apiWrapper` (`lib/api/apiWrapper.ts`) catches `ProjectNotFound` and
returns 404 `{ message }`, so routes and helpers may throw it freely without per-route
try/catch. The existing route-level resolution in Task 6/7 routes stays as-is (structural
inconsistency already adjudicated "functionally safe" in the M2 final review).

Rejected alternatives: route-level resolution everywhere (~24 blocks of boilerplate, and the
storage family's choke-point helper would still need threading); minimal credential-only fix
(leaves wrong-stack proxying; rejected by scope decision).

### 4.2 Registry extension: analytics columns

- `platform.projects` gains two **nullable** columns:
  - `logflare_url text` â€” base URL of the project's Logflare/analytics service.
  - `logflare_token_enc text` â€” private access token, encrypted with `PLATFORM_ENCRYPTION_KEY`.
- New migration `docker/volumes/platform/migrations/03-analytics.sql` using
  `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` (idempotent; correct on both fresh init, where
  01â†’02â†’03 run in order, and existing deployments applying it manually).
- **Graceful degradation (M2â†’M2.1 code-before-migration):** `projects.ts` retries its SELECT
  without the new columns when Postgres reports `column ... does not exist`, treats them as
  NULL, and logs an operator hint to apply `03-analytics.sql`. Mirrors the M2 I3
  missing-tableâ†’miss precedent; only the specific missing-column error triggers the retry.
- `resolve-connection.ts`: `ResolvedConnection` gains `logflareUrl: string | null` and
  `logflareToken: string | null` (decrypted at resolve time, same as other secrets). The
  global-env fallback populates them from `LOGFLARE_URL` / `LOGFLARE_PRIVATE_ACCESS_TOKEN`.
- **register CLI** (`docker/scripts/platform/register-project.ts`): new optional flags
  `--logflare-url` / `--logflare-token`; `--from-current-env` picks up `LOGFLARE_URL` +
  `LOGFLARE_PRIVATE_ACCESS_TOKEN` when present. Not added to the required-fields set.
- **Upgrade note (README):** rows registered under M2 (including the existing `default` and
  `proj-b`) have NULL analytics fields after migration â†’ their Logs pages return "not
  configured" (honest, not broken) until the operator backfills. README documents both paths:
  re-run the register CLI with `--from-current-env`, or a one-line
  `UPDATE platform.projects SET logflare_url = ..., logflare_token_enc = ...` example produced
  by a documented encrypt helper invocation.

### 4.3 Admin-client factory (storage 14 + auth users 3)

`lib/api/self-hosted-admin.ts` (today: single global Proxy client built from
`SUPABASE_URL`/`SUPABASE_SERVICE_KEY` at `:19`) gains:

```
getAdminClientForRef(ref: string): Promise<SupabaseClient>
```

- `IS_SELF_PLATFORM` true â†’ `resolveProjectConnection(ref)` â†’
  `createClient(conn.supabaseUrl, conn.serviceKey)` (kong URL fronts auth/storage paths).
  **No caching**: clients are created per call â€” cheap, and avoids stale credentials after a
  registry row update.
- `IS_SELF_PLATFORM` false â†’ returns the existing global client (zero-break; the existing
  `selfHostedSupabaseAdmin` export remains untouched for any other consumers).
- All 14 storage routes and the 3 `auth/[ref]/users/**` routes switch from the global import to
  the factory, passing their `ref` param. `ProjectNotFound` propagates to apiWrapper's 404.

The storage `sign.ts` / `sign-multi.ts` / `public-url.ts` host rewrite currently uses global
`SUPABASE_PUBLIC_URL`; per-ref, the public base is the resolved `supabaseUrl` (registry
`kong_url` is the client-reachable URL by registration contract); plain self-hosted keeps
`SUPABASE_PUBLIC_URL` unchanged.

### 4.4 Auth link routes (invite / magiclink / otp / recover)

Each inlines `${process.env.SUPABASE_URL}/auth/v1/<action>` + `Bearer SUPABASE_SERVICE_KEY`.
Per-ref: in self-platform mode resolve once at the top of the handler and use
`conn.supabaseUrl` + `conn.serviceKey`; plain self-hosted path byte-identical. Four small
same-shaped edits (no new helper needed beyond the resolver â€” the URL join is one template
literal; a shared helper would abstract four lines).

### 4.5 Analytics routes (3)

- New shared target helper (in `lib/api/self-hosted/logs.ts`):
  `getAnalyticsTarget(ref)` â†’ `{ baseUrl, token }`.
  - **URL semantics:** `logflare_url` stores the base URL (the per-project equivalent of
    `LOGFLARE_URL`, no path suffix); the helper appends the `/api/` segment for both branches,
    replicating today's `PROJECT_ANALYTICS_URL = ${LOGFLARE_URL}/api/` construction in one
    place.
  - Self-platform, resolved row: `conn.logflareUrl`/`conn.logflareToken`; **either NULL â†’ throw
    `AnalyticsNotConfigured`**, mapped to 404
    `{ message: 'Analytics is not configured for this project' }` (strict no-fallback).
  - Self-platform, unregistered `default` / plain self-hosted: global `LOGFLARE_URL` +
    `LOGFLARE_PRIVATE_ACCESS_TOKEN` (today's values, via the same `/api/` construction).
- `retrieveAnalyticsData` and both `log-drains*` routes consume the helper (removing their
  inline env reads).
- **Documented assumption:** a per-project analytics backend is a vanilla self-hosted stack
  whose Logflare identifies its project as `default`; the forwarded `?project=` param for
  resolved targets is therefore `default`. Revisit only if a registered stack diverges.

### 4.6 Credential-bearing routes

**`temporary.ts`** â€” self-platform mode mints a short-lived HS256 JWT instead of returning a
permanent key (matches cloud semantics):

- New `lib/api/self-platform/mint-jwt.ts`: hand-rolled HS256 via `node:crypto` `createHmac`
  (~15 lines; signing only, no verification; no new dependency â€”
  `jsonwebtoken`/`jose` exist nowhere in the workspace and adding one was rejected).
- Payload: `role` parsed from the request's `claims` JSON, allowlisted to
  `anon | authenticated | service_role` (400 otherwise); `exp` from `authorization_exp`
  (integer seconds, default 300, clamped to [60, 3600], invalid â†’ 400); `iat` now.
- Secret: `conn.jwtSecret` for the resolved ref. Empty/missing secret â†’ **fail closed 500**
  (never fall back to returning a key).
- Response shape unchanged: `{ api_key: <jwt> }` â€” GoTrue/Realtime/PostgREST verify against
  the same stack secret, so the Realtime Inspector keeps working.
- Plain self-hosted: byte-identical to today (returns `SUPABASE_SERVICE_KEY ?? ''`).

**`props/project/[ref]/api.ts`** â€” self-platform mode maps from the resolver: endpoint/host
fields follow the M2 C1 convention (`new URL(conn.supabaseUrl).host`, protocol from the URL),
`defaultApiKey = conn.anonKey`, `serviceApiKey = conn.serviceKey`, `restUrl = conn.restUrl`,
db fields from the row. No live consumer today, but the endpoint stays contract-shaped in case
upstream revives one. Plain self-hosted byte-identical.

**`props/project/[ref]/index.ts`** â€” same treatment for its metadata-only payload
(project name/ref/db fields from the resolver; empty `services` array kept).

### 4.7 pg-meta lints + migrations threading

The two routes gain the Task 8 treatment: pass `projectRef` into `executeQuery` so resolved
refs run against their own database. Route-level behavior for unknown refs matches the query
route (404 via the same mapping). `mcp.ts` explicitly untouched.

### 4.8 Real pagination + api-types alignment

- Data layer (`projects.ts`): `listAllProjects(limit, offset)` / `listProjectsByOrgId(orgId,
limit, offset)` add SQL `LIMIT/OFFSET` (parameterized); new `countAllProjects()` /
  `countProjectsByOrgId(orgId)` return `COUNT(*)`. `pagination.count` = total rows (per
  api-types docs), not page size.
- `list-user-projects.ts`: bespoke `OrgProjectListItem`/`GlobalProjectListItem`/`Pagination`
  interfaces are deleted; return types become
  `components['schemas']['OrganizationProjectsResponse']` and
  `components['schemas']['ListProjectsPaginatedResponse']` (`satisfies` style per the Task 7
  precedent). Required fields the registry cannot source use honest constants
  (`is_branch_enabled: false`, `is_physical_backups_enabled: null`, `subscription_id: null`,
  placeholder `inserted_at` retained; org-list `databases[].cloud_provider` from the row).
- Empty-registry fallback (single `DEFAULT_PROJECT`) is preserved; with `offset >= 1` the
  fallback page is empty while `count` stays 1 (consistent with total-count semantics).
- Route-level validation: `limit`/`offset` must parse as non-negative integers; NaN, negatives,
  or array-valued params â†’ 400. Ledger #7: array-valued `slug` â†’ 400 instead of
  `String(...)`-coerced 404.

### 4.9 Error handling summary

| Condition                                   | Behavior                                                                                                                        |
| ------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------- |
| Unknown non-default ref (any route)         | `ProjectNotFound` thrown â†’ apiWrapper maps to 404 `{ message }` (new, tested; existing 500 catchall unchanged for other errors) |
| Resolved row, analytics fields NULL         | 404 "Analytics is not configured for this project" â€” never default-stack data                                                   |
| M2.1 code, migration 03 not applied         | SELECT retry without new columns; fields NULL; operator log; everything else works                                              |
| `temporary.ts` empty jwtSecret              | 500 fail-closed                                                                                                                 |
| Invalid `claims` role / `authorization_exp` | 400                                                                                                                             |
| Invalid limit/offset/array params           | 400                                                                                                                             |

## 5. Testing strategy

The three route families currently have **zero tests**. M2.1 adds:

- **Route handler tests** for every touched route (vi.hoisted env-injection pattern):
  resolved-ref behavior, unknown-ref 404, method 405s where cheap.
- **Sibling `*.self-hosted.test.ts` zero-break guards** for every touched route
  (dynamic-import pattern) asserting plain-self-hosted behavior unchanged â€” including
  byte-identical responses for `temporary.ts` and both props routes.
- **Helper unit tests:** admin-client factory (3 modes: resolved / unregistered-default /
  plain); `mint-jwt` (signature verifiable with the secret, exp/iat/role claims, clamping,
  allowlist, empty-secret failure); `getAnalyticsTarget` (resolved / NULLâ†’throw / fallback);
  `projects.ts` missing-column retry (error-message triggered, other errors propagate);
  pagination (LIMIT/OFFSET parameterization, count queries, fallback-page edge).
- **register CLI tests** extended for the new flags and env pickup.
- **E2E acceptance** on the two-project environment (default + proj-b, per resolver README):
  auth users list, storage buckets, Logs page, Realtime Inspector token mint â€” per project;
  plus bidirectional plain self-hosted regression. Analytics backfill exercised on `default`.
- **Ledger #12 closure:** `pnpm typecheck` and `pnpm lint --filter=studio` run with recorded
  output as part of final acceptance.

## 7. Acceptance criteria

1. With `proj-b` registered pointing at a distinct stack (or the shared stack in the current
   test env), every `[ref]` route family resolves per ref; no route reads global
   `SUPABASE_URL`/`SUPABASE_SERVICE_KEY`/`LOGFLARE_*` for a resolved ref.

   Realtime Inspector works against both projects.
3. No client-facing route returns another project's (or the global) keys for a resolved ref.
4. Project lists paginate correctly (`count` = total; slicing real) and typecheck against
   api-types with no bespoke list interfaces left.
5. Full studio unit suite green including new sibling zero-break guards; plain self-hosted
   byte-identical on all touched routes; typecheck + lint clean (output recorded).
6. M2â†’M2.1 upgrade path verified: pre-migration code degrades gracefully; post-migration
   backfill restores Logs on `default`.
