# M0.4 Write Fence and Final Provider Contracts

## Result

PASS on 2026-07-12 using the customized `deluxebear/postgres:17` image and
three independently running application clients representing the routed data
plane, a pooler, and a direct PostgreSQL connection.

The finalized contracts are implemented and unit tested in
`apps/backup-operator/internal/contracts/providers.go`.

## Write-fence proof

The destructive spike is
`apps/backup-operator/spikes/write-fence.sh`. It applies two required layers:

1. stop routed data-plane and pooler paths;
2. fence the database role used by direct clients, revoke connection access,
   resolve prepared transactions, and terminate existing application sessions.

The Agent retains a local PostgreSQL socket control channel so it can continue
verification and recovery while all application writers are blocked.

Observed result:

```text
data_plane_state=exited
pooler_state=exited
direct_client_state=running
active_app_sessions_before=3
active_app_sessions_after=0
prepared_transactions_before=1
prepared_transactions_after=0
direct_login_blocked=true
control_socket_sql=1
rows_at_fence=3
rows_after_release=4
RESULT=PASS
```

The direct client container intentionally remains running. Its inability to
reconnect proves that stopping Kong or a pooler is not the direct-connection
fence. No rows are added during the fenced interval. After an explicit release,
the direct client reconnects and writes again. Stopped data-plane components
are not implicitly restarted by fence release.

## Binding fence evidence

Restore may continue only when fresh evidence proves all of the following:

- routed data-plane paths are blocked;
- poolers are blocked;
- direct application login is blocked at PostgreSQL;
- active application writers are zero;
- prepared transactions are zero;
- alternate writable primaries are zero;
- the Agent control channel remains healthy.

Missing, expired, or uncertain evidence fails closed. Patroni pause and
Kubernetes Service cutover are coordination mechanisms, not write fences.

## Final contract boundaries

### ControlStore

- Must be outside the managed recovery domain.
- Persists operation state and outgoing task in one transaction.
- Deduplicates task results.
- Never uses the managed Supabase database as restore authority.

### TopologyProvider

- Returns fresh, typed observations with exactly one reachable primary and one
  PostgreSQL system identifier.
- Models static, Patroni, self-managed Kubernetes, and optional managed-operator
  ownership explicitly.
- Rebuilds standbys only through the recognized topology authority.

### BackupProvider

- Binds provider, repository, stanza, system identifier, and database history.
- Exposes typed capability/inspection/backup/restore operations.
- The default implementation is the pgBackRest already packaged in the
  customized Supabase PostgreSQL image.
- Restore can require a read-only repository view.

### WriteFenceProvider

- Separately engages, verifies, and releases a fence.
- Returns an expiring handle and complete typed evidence.
- Does not equate process shutdown, Patroni pause, or routing changes with a
  verified database fence.

### RecoveryStrategy

- Supports in-place, replacement-workload, and optional managed-replacement
  modes without inferring one from deployment type.
- Replacement-workload recovery requires a new destination/PVC.
- Controller-owned storage cannot be restored in place.
- Execution consumes a verified fence handle; rollback is a distinct operation.

## M0 architecture pins

1. Independent PostgreSQL and external SQLite are valid control-store modes;
   the target database and shared data volumes are rejected.
2. Patroni pause is not fencing. All old standbys must be rebuilt after PITR.
3. Default Kubernetes recovery uses the customized image, built-in pgBackRest,
   a replacement StatefulSet, and new PVCs.
4. CloudNativePG/CNPG-I is optional and does not block the core release.
5. OrioleDB17 remains disabled until its published image passes the managed
   pgBackRest capability preflight and destructive recovery matrix.
6. A complete write fence must cover routed services, poolers, direct clients,
   active writers, prepared transactions, and alternate primaries.
