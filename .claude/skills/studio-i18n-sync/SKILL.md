# Studio i18n: upstream sync & translation

## Architecture (read this first)

- **English source string IS the translation key.** `$t('Save changes')` looks up
  `zh-CN.json`; a missing key falls back to the English key itself. There is no `en.json`.
- **`apps/studio/lib/i18n/locales/zh-CN.json` is the only hand-valuable artifact.**
  Upstream never touches it, so translations are structurally safe across merges.
- **All source wrapping is regenerable.** `scripts/i18n/wrap.ts` (idempotent ts-morph
  codemod) wraps JSX text / allowlisted attrs / sonner toasts as `$t('...')` and injects
  `import { t as $t } from '@/lib/i18n'`. The `$t` alias exists because components often
  bind a local `t` (e.g. `.map((t) => ...)`) — a bare `t` import collides.
- Mechanics reference: `apps/studio/scripts/i18n/README.md`. Design/plan docs:
  `docs/superpowers/specs|plans/2026-07-02-studio-i18n-zh-cn*`.

## Workflow A — Sync with upstream

```bash
./apps/studio/scripts/i18n/sync-upstream.sh upstream/master
```

The script: verifies the ref → merges (no commit) → resolves conflicted `.tsx` under
`components/`/`pages/` by taking upstream (they get re-wrapped) → re-runs `wrap.ts`
(idempotent) → runs `translate.ts` only if `I18N_TRANSLATE_ENDPOINT`/`API_KEY` are set,
otherwise warns and skips → verifies the activation glue → stages results.

Know these two deliberate behaviors:

1. **No translation credentials exist in this repo.** The original catalog was produced
   by LLM subagents (Workflow B), not `translate.ts`. After a sync, new upstream strings
   are English until you run Workflow B (or configure `I18N_TRANSLATE_*`). English
   fallback means nothing is broken in the meantime.
2. **Two "activation glue" files are excluded from take-theirs** and left as manual
   conflicts: `pages/_app.tsx` (mounts `I18nProvider`) and
   `components/interfaces/UserDropdown.tsx` (mounts `LanguageSwitcher`). They are
   hand-authored and NOT regenerated by the codemod — blindly taking upstream would
   silently deactivate zh-CN. Resolve keeping the mounts; the script grep-checks both
   and exits 1 if either is missing.

After the sync completes, run the verification gates (below), then Workflow B for the
new keys.

## Workflow B — Translate new keys via subagents

Use when new untranslated keys exist (after a sync, or after wrapping new components)
and no `I18N_TRANSLATE_*` credentials are available. All steps run with
`cwd = apps/studio`.

### 1. Split missing keys into batches

```bash
pnpm exec tsx scripts/i18n/batch.ts split          # default 150 keys/batch
```

Prints the workdir (default `$TMPDIR/studio-i18n-batches`) containing
`batch-NN.in.json` files. **Keep batches at ~150 keys** — at ~450, agents start
producing invalid JSON, miscounting, or giving up partway (observed 4/15 failures at
450 vs. near-zero at 150).

### 2. Dispatch one subagent per batch (parallel, haiku-class model)

File-in/file-out keeps translations out of your context. Prompt template per batch:

```
You are a professional English→Simplified Chinese (zh-CN) translator for the
Supabase Studio developer dashboard UI.

Read this JSON array of English UI strings: <workdir>/batch-NN.in.json

Output a JSON OBJECT mapping each ORIGINAL English string → its zh-CN translation.
Rules:
- Keys must be the input strings copied BYTE-FOR-BYTE (curly quotes ’ “ ” and all
  punctuation exactly as-is — do not straighten quotes). Every input element appears
  exactly once; do not add, drop, or merge keys.
- Preserve {{placeholders}} verbatim.
- Keep product/tech names in English: Supabase, PostgreSQL, SQL, API, RLS, JWT,
  OAuth, URL, CLI, JSON, GraphQL, Webhook, Edge Function, etc.
- If a string is purely a product name / code / identifier / unit that should stay
  English, set its value EQUAL to the key (it will be treated as "kept English").
- Natural, concise, professional Chinese. Output MUST be valid JSON.

Write ONLY the JSON object to (overwrite): <workdir>/batch-NN.out.json
Then re-read and parse it; confirm key count equals input count.
Report ONLY: "batch-NN: <N> keys, verified" or the specific problem.
Do not paste the translations.
```

### 3. Validate, merge, iterate

```bash
pnpm exec tsx scripts/i18n/batch.ts check   # flags REDO batches (invalid/missing/mostly-English)
pnpm exec tsx scripts/i18n/batch.ts merge   # merges valid entries into zh-CN.json, writes remaining.json
```

`merge` is defensive by design — agents miscount, hallucinate keys, and straighten
curly quotes, so it only accepts keys that exist verbatim in `keys.json`, ignores
`value === key` entries, never deletes existing translations, and writes sorted output.
Re-run split/dispatch on `remaining.json` leftovers with smaller batches if needed.

**Do not chase 100% coverage.** Remaining keys that are product names, code snippets,
units, or cert headers (`GitHub`, `ALTER ROLE`, `100 GB`, …) are _correct_ as English
fallback — leaving them out of the catalog is the intended end state (~98% is "done").

### 4. Commit

```bash
git add apps/studio/lib/i18n/locales/zh-CN.json
git add -f apps/studio/scripts/i18n/keys.json   # -f: keys.json is gitignored repo-wide
```

## Verification gates

Run after any sync or re-wrap, in this order (cheapest first):

```bash
# 1. Focused unit tests — NEVER `pnpm --filter studio test`: its --coverage flag
#    instruments the whole repo and hangs >10 minutes. Always use vitest run directly:
pnpm --filter studio exec vitest run lib/i18n scripts/i18n

# 2. Typecheck (catches malformed wraps, $t collisions):
pnpm --filter studio exec tsc --noEmit -p tsconfig.json

# 3. Production build — REQUIRED after any large re-wrap. The build catches what tsc
#    cannot (e.g. a 'use client' directive displaced below an injected import):
pnpm build --filter=studio
```

If prettier is needed after a re-wrap, format **only the changed files**
(`git diff --name-only ... | xargs pnpm exec prettier --write`) — a blanket format of
components/pages creates massive unrelated diff noise against upstream.

## Gotchas (each cost real debugging time)

- **Codemod bugs surface at build, not typecheck.** Two historical examples: imports
  inserted at index 0 broke `'use client'` prologues (build-only failure); bare `t`
  collided with local params (2 type errors out of 1,337 files, plus silent-wrong-call
  risk). If you modify `transform.ts`, re-run the full gate sequence.
- **Multiline JSX text is collapsed** (`\s+` → single space) before becoming a key, so
  keys match React's rendered text and never contain raw newlines. Don't "fix" this.
- **Fragment keys are a known limitation**: JSX expressions split sentences into
  fragments (`"be undone"`, `"with cascade?"`). Accepted for machine translation;
  don't try to stitch them.
- **`SupportFormPage.test.tsx` times out under full-suite load** but passes in
  isolation. Verify a suspected flake by running the file alone before investigating.
- **Language switching remounts the whole tree** (`key={locale}` in `I18nProvider`) —
  intentional, since the global `$t` binding must re-read. Rare action, cheap.

## Adding a new language

1. Create `apps/studio/lib/i18n/locales/<locale>.json` (start with `{}`).
2. Add the locale to `LOCALES` and `resources` in `apps/studio/lib/i18n/index.ts`.
3. Add its label to `LOCALE_LABELS` in `apps/studio/components/ui/LanguageSwitcher.tsx`.
4. Run Workflow B with the target language substituted in the prompt template
   (`batch.ts` currently reads/writes `zh-CN.json`; point `CATALOG_PATH` at the new
   catalog or generalize it with a `--locale` flag first).
